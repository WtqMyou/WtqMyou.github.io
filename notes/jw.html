计算机网络原理 ​

计算机网络和因特网 ​

什么是因特网 ​

回答“什么是因特网”可以通过两种不同的方式：第一种是描述其具体构成（硬件和软件组件），第二种是从为分布式应用提供服务的基础设施的角度来描述。

具体构成描述 ​

因特网是一个世界范围的计算机网络，互联了遍及全世界的计算设备。

核心概念: 因特网的基本硬件和软件组件

  • 主机 (host) / 端系统 (end system)：与因特网相连的设备（如PC、服务器、智能手机及非传统的物联网设备等）。它们容纳并运行应用程序。

  • 通信链路 (communication link)：端系统通过不同类型的物理媒介（同轴电缆、双绞铜线、光纤、无线电频谱）相互连接。链路的传输速率以 比特/秒 (bps) 度量。

  • 分组 (packet)：发送端系统将要发送的数据分段，并加上首部字节后形成的信息包。

  • 分组交换机 (packet switch)：负责从输入链路接收分组并从输出链路转发分组。最常见的两种类型是：

    • 路由器 (router)：通常用于网络核心。
    • 链路层交换机 (link-layer switch)：通常用于接入网中。

  • 因特网服务提供商 (ISP)：如本地有线公司、蜂窝数据ISP等。端系统通过ISP接入因特网，各ISP网络也必须彼此互联。

  • 协议 (protocol)：控制因特网中信息的接收和发送，最重要的两个协议是 TCP (传输控制协议) 和 IP (网际协议)。

易错辨析: 标准的制定

因特网标准的制定对于保证各系统和产品的交互操作至关重要。因特网标准由 IETF (因特网工程任务组) 研发，其标准文档被称为 RFC (请求评论)。

服务描述 ​

从完全不同的角度来看，因特网是为应用程序提供服务的基础设施。

核心概念: 基于服务视角的因特网

  • 分布式应用 (distributed application)：涉及多个相互交换数据的端系统的因特网应用（如Web、即时通信、流媒体等）。这类应用运行在端系统上，而不在网络核心的分组交换机上运行。
  • 套接字接口 (socket interface)：因特网为端系统提供的编程接口，它规定了运行在一个端系统上的程序如何请求因特网基础设施向另一个端系统交付数据。

易错辨析: 套接字接口的类比

就像使用邮政服务寄信必须遵循一定的规则（信封、地址、邮票、投入信箱）一样，网络应用发送数据也必须遵循套接字接口规定的规则，因特网才能成功交付数据。

什么是协议 ​

核心概念: 协议的严格定义

协议 (protocol) 定义了在两个或多个通信实体之间交换的报文的格式和顺序，以及报文发送/接收或其他事件所采取的操作。

无论是人类活动（如“你好”、“几点了”）还是网络通信（如 TCP的连接请求与响应、Web文档的获取），报文的交换和根据报文所采取的操作都是协议的核心。

网络边缘 ​

通常把与因特网相连的计算机和其他设备称为端系统 (end system)，因为它们位于因特网的边缘。端系统有时也被进一步划分为 客户 (client) 和 服务器 (server)。

接入网 ​

接入网 (access network) 是指将端系统物理连接到其边缘路由器 (edge router) 的网络。边缘路由器是端系统去往其他远程端系统路径上的第一台路由器。

核心概念: 常见接入网技术

  • 家庭接入：

    • 数字用户线 (DSL)：利用电话公司现有的本地电话基础设施。通过 频分复用 技术（如 0-4kHz为电话，4kHz-50kHz为上行数据，50kHz-1MHz为下行数据），数据和电话信号共享同一链路。本地中心局的 DSLAM 将二者分离。传输速率通常是不对称的。
    • 电缆因特网接入 (Cable)：利用有线电视公司的基础设施。光缆和同轴电缆结合形成 HFC (混合光纤同轴) 系统。头端的 CMTS 类似于 DSLAM。
    • 光纤到户 (FTTH)：从本地中心局直接到家庭的光纤路径。分为 有源光纤网络 (AON) 和 无源光纤网络 (PON)。
    • 5G固定式无线：使用波束成形技术，数据以无线方式从基站发送到家中的调制解调器。

  • 企业和家庭局域网接入：

    • 以太网 (Ethernet)：用户使用双绞铜线与以太网交换机相连，是最为流行的局域网技术。
    • WiFi (802.11 无线局域网)：用户通过无线接入点(AP)发送和接收分组，广泛应用于家庭、大学和咖啡馆。

  • 广域移动无线接入：

    • 如 3G、4G LTE 和 5G，通过蜂窝网提供商运营的基站进行大范围内的无线通信。

易错辨析: DSL与电缆(HFC)的本质区别

DSL 是一种点对点专用的接入方式（每户一条独立的线连到本地中心局的DSLAM）；而 电缆接入 是一种共享广播媒体（由头端发送的分组下行经每段链路到达每个家庭）。因此，电缆接入的实际速率极大地受到并发活跃用户数量的影响，且需要多路访问协议来协调上行传输以避免碰撞。

物理媒介 ​

为了将一个比特从一个端系统传输到另一个端系统，必须通过电磁波或光脉冲跨越物理媒介 (physical medium)。物理媒介分为两大类：导引型媒介 (guided media)（如光缆、双绞线、同轴电缆，电波沿着固体媒介前行）和 非导引型媒介 (unguided media)（如无线电，电波在空气或外层空间中传播）。

核心概念: 常用物理媒介

  • 双绞铜线 (UTP)：最便宜、最常用的导引型传输媒介，由两根螺旋状绝缘铜线组成，用于减少电气干扰。广泛用于以太网局域网。
  • 同轴电缆：由两根同心铜导体组成，主要用于电缆电视和电缆因特网接入，可被用作导引型共享媒介。
  • 光纤 (Fiber optics)：导引光脉冲的柔软媒介，每个脉冲表示一个比特。支持极高比特速率，极低信号衰减，且不受电磁干扰，常用于跨海链路和因特网主干。
  • 陆地无线电信道：通过电磁频谱承载信号，分为短距离（蓝牙/WiFi）、局域和广域（蜂窝网络）无线信道。极度依赖传播环境（路径损耗、遮挡、多径衰落、干扰）。
  • 卫星无线电信道：包含 同步卫星 (Geostationary)（传播时延高，约280ms）和 近地轨道卫星 (LEO)。

网络核心 ​

网络核心 (network core) 是由互联因特网端系统的分组交换机和链路构成的网状网络。在这个网状网络中，数据是如何通过网络进行交换和传输的？主要有两种基本方法：分组交换 和 电路交换。

分组交换 ​

在各种网络应用中，端系统彼此交换报文 (message)。为了从源端系统向目的端系统发送一个报文，源端系统将长报文划分为较小的数据块，我们称之为 分组 (packet)。

核心概念: 存储转发机制 (Store-and-Forward Transmission)

在分组交换网络中，大多数分组交换机（路由器或链路层交换机）在链路的输入端采用存储转发传输机制。这意味着：交换机必须接收到整个分组的全部比特后，才能开始向输出链路传输该分组的第一个比特。

每台分组交换机都有多条连接到它的链路。对于每条连接的链路，该交换机都具有一个输出缓存 (output buffer)（也称为输出队列），用于存储路由器准备发往该链路的分组。

  • 排队时延 (Queuing delay)：如果到达的分组发现该链路正忙于传输其他分组，它必须在输出缓存中等待。
  • 丢包 (Packet loss)：因为缓存空间是有限的，如果一个到达的分组发现缓存已满，路由器将不得不丢弃该分组，这就是网络拥塞导致的丢包现象。

易错辨析: 路由选择与转发的区别

这二者共同构成了网络层最核心的两大功能，但含义截然不同：

  • 路由选择 (Routing)：属于全局控制层面的操作。网络依靠 路由选择协议 来决定分组从源到目的地的端到端路径，并计算出相应的转发表 (forwarding table)。
  • 转发 (Forwarding)：属于局部数据平面的操作。当分组到达路由器时，路由器提取首部中的目的地址，查询转发表，将该分组从路由器的输入接口转移到正确的输出接口。

电路交换 ​

传统的电话网络就是典型的**电路交换 (circuit switching)**网络。

核心概念: 电路交换的核心思想

在端系统通信期间，通信线路上所需要的资源（如缓存、链路传输速率）会被预留。无论这些线路是否真正在传输数据，预留的链路带宽将被这一个连接（称为一个 电路 (circuit)）独占，直到通信结束才释放。

核心概念: 电路交换中的复用技术

一条链路通常被多个连接共享，电路交换主要使用两种复用技术：

  • 频分复用 (FDM, Frequency-Division Multiplexing)：将物理链路的频谱带宽划分为多个独立的频段，每个连接在其持续期间专用的占据一个频段（类似无线电广播频率）。
  • 时分复用 (TDM, Time-Division Multiplexing)：将时间划分为固定长度的 帧 (frame)，每个帧进一步划分为固定数量的 时隙 (slot)。网络在每个帧中为连接指定一个固定的时隙，专门用于传输该连接的数据。

易错辨析: 分组交换 vs. 电路交换：孰优孰劣？

这是一个计算机网络中极为经典的辨析题：

  • 资源利用率：分组交换不预留资源，而是采用了统计多路复用 (Statistical Multiplexing)，按需共享带宽。由于用户通常不是持续发送数据的（突发性数据），分组交换的整体链路利用率远高于电路交换。
  • 性能保证：电路交换预留了带宽，可以提供有保证的恒定传输速率，完全没有排队时延；而分组交换由于无法预知突发流量，容易在高峰期导致严重的排队时延甚至丢包。
  • 结论：今天的因特网全面采用分组交换，但在面临实时性要求极高的应用时依然面临挑战。

网络的网络 ​

因特网的核心并不是单一的整体，而是由众多的网络相互连接形成的，即“网络的网络”。

核心概念: 因特网的层次结构

端系统通过 接入ISP (Access ISP)（如本地电信运营商）连入因特网。为了让世界上任何人都能互相通信，这些接入ISP必须彼此相连。这形成了一个极其复杂的商业和拓扑结构：

  • 第一层ISP (Tier-1 ISP)：如AT&T、Sprint、NTT等，它们拥有覆盖全球的骨干网，并与所有其他第一层ISP直接相连。
  • 区域ISP (Regional ISP)：它们连接着接入ISP，并向上连接到第一层ISP。
  • 存在点 (PoP, Point of Presence)：ISP网络中供其他较底层的ISP接入的路由器群组。
  • 多宿主 (Multi-homing)：任何ISP（除了第一层外）都可以连接到两个或多个提供商ISP，以保证网络的高可用性和抗灾性。
  • 对等 (Peering)：同等层级的两个ISP为了避免向上一级ISP支付流量费，直接将它们的网络连接起来进行流量交换。
  • 因特网交换点 (IXP, Internet Exchange Point)：通常是一个独立的物理建筑内包含的一台或多台交换机，允许多个ISP在其中共同对等互联。
  • 内容提供商网络 (Content Provider Network)：如谷歌网络 (Google Network)。大型科技巨头建立自己的私有全球网络设施，将其数据中心通过专线互联，并尽可能多地绕过第一层/区域ISP，直接在底层与接入ISP对等互连，从而节省费用并提供更低延迟的服务。

协议层次及其服务模型 ​

复杂的系统通常采用分层结构，网络架构也不例外。通过分层，我们可以将庞大而复杂的网络系统分解为若干个更小、功能更加单一的组件。

分层体系结构 ​

核心概念: 协议分层与服务模型

为了给复杂系统的设计提供结构，网络设计者以分层 (layer) 的方式组织协议以及实现这些协议的网络硬件和软件。 每个协议属于这其中的某一层。每一层通过在该层中执行某些内部动作，并使用直接下层提供的服务，来向它的上一层提供服务 (service)。一个协议层能够用软件、硬件或者两者的结合来实现。 各层的所有协议组合在一起，就被称为协议栈 (protocol stack)。

易错辨析: 分层架构的优缺点辨析

分层架构是计算机科学中最经典的设计模式之一，但它并非完美无缺：

  • 优点：模块化。分层提供了一种结构化方式来讨论系统组件，极大地方便了系统的维护与更新。如果某一层提供的服务和接口保持不变，该层的具体实现可以任意更改，而完全不会影响系统的其他层。
  • 缺点：一方面，某一层可能冗余地执行与另一层相同的任务（例如，链路层和运输层可能都提供错误恢复功能）。另一方面，某层的功能可能需要仅在其他层才出现的信息，这在一定程度上打破了分层的严格界限，导致跨层开销。

因特网协议栈 ​

因特网的协议栈由5个层次组成。我们在学习和排查网络故障时，通常习惯从上向下（自顶向下）或者从下向上（自底向上）分析。

核心概念: 因特网五层协议栈

  1. 应用层 (Application Layer)：网络应用程序及它们的应用层协议存留的地方。包括HTTP（Web文档）、SMTP（电子邮件）、FTP（文件传送）和DNS（域名系统）。应用层产生的信息分组称为报文 (message)。
  2. 运输层 (Transport Layer)：负责在应用程序端点之间传送应用层报文。因特网主要有两种运输协议：TCP（提供面向连接、可靠交付和拥塞控制）和UDP（提供无连接的、尽力而为的服务）。运输层的分组称为报文段 (segment)。
  3. 网络层 (Network Layer)：负责将称为数据报 (datagram) 的网络层分组从一台主机移动到另一台主机。它包含了著名的IP协议以及众多的路由选择协议，是整个互联网的神经中枢。
  4. 链路层 (Link Layer)：为了将分组从一个节点移动到路径上的下一个节点，网络层必须依赖链路层的服务。常见协议包括以太网 (Ethernet)、WiFi和电缆接入网协议。链路层的分组称为帧 (frame)。
  5. 物理层 (Physical Layer)：负责将帧中的每一个比特从一个节点通过物理媒介（如双绞线、光纤、无线电波）移动到下一个节点。

OSI 参考模型 ​

易错辨析: OSI七层模型与因特网五层模型的差异

国际标准化组织（ISO）曾经提出过一个七层的 OSI（开放系统互连）参考模型。它在因特网五层模型的基础上，在应用层和运输层之间增加了两层：

  • 表示层 (Presentation Layer)：使通信的应用程序能够解释交换数据的含义（如数据压缩、加密、数据描述等）。
  • 会话层 (Session Layer)：提供了数据交换定界和同步功能（如建立检查点和数据恢复方案）。

为什么因特网抛弃了这两层？ 因特网的设计者认为，这些服务如果需要，应该由应用程序开发者自己实现在应用层中，而不是强制网络底层架构去提供这些可能会带来额外负担的服务。

封装 ​

核心概念: 封装与解封装 (Encapsulation & Decapsulation)

数据在网络中传输时，必须经历“打包”和“拆包”的过程：

  • 封装：在发送端，数据沿着协议栈向下传递。在每一层，上一层交下来的分组被当做本层的有效载荷 (payload)，并在其上附加本层的首部字段 (header)，从而组合形成本层的新分组。
  • 解封装：在接收端，数据沿着协议栈向上传递。每一层检查并剥离掉属于本层的首部，将剩下的有效载荷向上交付给上一层。

例题: 端到端的数据封装实战分析

假设主机A通过中间的一台路由器向主机B发送一封电子邮件，在这个过程中：

  1. 源主机A：邮件客户端产生应用层报文 。向下传给TCP，TCP加上首部 形成报文段 。向下传给网络层IP，IP加上首部 形成数据报 。最后以太网链路层加上首部 形成帧 并发往物理链路。
  2. 路由器：由于路由器是网络层设备，它通常只实现底部的三层。它从物理层接收信号，在链路层提取出数据报，将网络层IP首部 拆开读取目的IP地址以决定转发接口，然后再重新加上新的链路层首部转发出去。
  3. 目的主机B：接收到帧后，从底向上逐层检查并剥离 , , ，最终将无损的邮件内容 交付给主机B的邮件服务器软件。

面对受攻击的网络 ​

因特网最初的设计是一群互相信任的研究人员为了共享资源而建立的，并没有充分考虑安全性。这导致当今的因特网极易受到各种各样的恶意攻击。

核心概念: 恶意软件与僵尸网络

恶意软件 (Malware) 可以进入并破坏端系统设备，最常见的两类是：

  • 病毒 (Virus)：需要某种形式的人类交互才能感染用户设备的恶意软件（例如诱骗用户打开包含恶意代码的电子邮件附件）。
  • 蠕虫 (Worm)：不需要任何明显的人类交互就能自行在网络中扫描漏洞并进入设备的恶意软件。

被恶意软件感染的设备通常会被黑客在后台偷偷招募进一个庞大的网络，称为僵尸网络 (Botnet)。黑客可以远程统一控制这些僵尸主机，用来发送海量垃圾邮件或发动大规模网络攻击。

易错辨析: 拒绝服务攻击 (DoS) 的三种主要类型

DoS (Denial of Service) 攻击的目的是向目标发送海量垃圾数据或特殊指令，使得网络、主机或其他基础设施无法为合法用户提供服务。

  1. 弱点攻击 (Vulnerability attack)：向目标主机上运行的脆弱的应用程序或操作系统发送制作精细的报文，导致系统停止运行或直接崩溃。
  2. 带宽泛洪 (Bandwidth flooding)：向目标发送极大数量的分组，使得目标的接入链路严重拥塞，导致合法分组被堵死而无法到达服务器。
  3. 连接泛洪 (Connection flooding)：在目标主机中创建海量的半开或全开的TCP连接，使主机陷入由于维护伪造连接而耗尽系统资源（如内存和CPU）的困境。

现代黑客为了躲避追踪和增强破坏力，多采用 分布式拒绝服务 (DDoS) 攻击，即控制成千上万的僵尸主机同时向目标发送流量，这极大地增加了防守方过滤恶意流量的难度。

核心概念: 分组嗅探与IP伪造

  • 分组嗅探 (Packet Sniffing)：在广播环境（如公共WiFi或早期共享以太网）中，传输的信号会被范围内的所有设备接收。将网络接口设置为“混杂模式”的接收方可以记录流经它的每一个分组的副本。这可能导致明文传输的密码和私人信息泄露。防御的根本方法是使用密码学加密技术。
  • IP 伪造 (IP Spoofing)：将具有虚假源IP地址的分组注入因特网的过程。由于互联网早期的路由器通常只根据目的IP地址来转发分组，而不严格验证源IP的真伪，黑客可以借此伪装成受信任的设备。防御方法通常是端点认证和在网络边缘路由器配置严格的入口过滤机制。

计算机网络和因特网的历史 ​

了解计算机网络的发展历史，有助于我们理解各种协议和机制为何会演变成今天的样子。

核心概念: 因特网发展的几个关键阶段

  • 早期发展（1961-1972）：分组交换理论的提出。为了解决早期电话网络的电路交换在数据传输中效率低下的问题，科学家们提出了排队与分组交换的概念。此时诞生了最早的分组交换网 ARPAnet。
  • 专用网络与网络互联（1972-1980）：不同组织开始建立自己的专用计算机网络（如 ALOHAnet、Telenet 等）。为了让这些互不兼容的网络能够通信，网络互联 (internetworking) 的概念和 TCP/IP 协议 的雏形被提出，确立了端到端的复杂性留在边缘、网络核心保持简单的基本架构原则。
  • 网络的激增（1980-1990）：TCP/IP 被正式确立为 ARPAnet 的标准协议。同时，**DNS（域名系统）**被发明，用于将人类可读的因特网名字映射为32位的 IP 地址。以太网在这个时期由于廉价的同轴电缆和双绞线而迅速普及。
  • 因特网的爆炸（1990年代）：因特网迎来了最重要的应用——万维网 (World Wide Web)。HTML、HTTP 和早期浏览器的出现将因特网从研究人员专属工具变成了普通家庭日常使用的基础设施。
  • 新世纪（2000年至今）：宽带接入、无线网络 (WiFi、移动蜂窝网络)、云计算与大型内容提供商（如 Google, 微软）私有网络的崛起，以及物联网 (IoT) 设备的激增，构成了当今极度复杂的网络生态。

应用层 ​

应用层是计算机网络体系结构的最顶层，也是网络应用程序驻留的地方。它是我们日常接触最多、也是创新最活跃的一层。

应用层协议原理 ​

研发网络应用程序的核心是写出能够运行在不同端系统上，并且可以通过网络相互通信的程序。重点在于，开发人员不需要为网络核心设备（如路由器、链路层交换机）写软件，只需在端系统上进行开发即可。

网络应用程序体系结构 ​

在设计一个新的网络应用时，开发者首先需要决定采用哪种体系结构 (architecture)。现代网络应用主要采用以下两种主流体系结构之一。

核心概念: 客户-服务器体系结构 (Client-Server, C/S)

在该结构中，有一个总是打开的主机称为服务器 (Server)，它服务于许多来自其他称为客户 (Client) 的主机的请求。

  • 服务器的特征：拥有固定的、周知的 IP 地址，始终在线等待请求。为了应对海量请求，通常依靠由大量服务器组成的数据中心 (Data Center) 来进行扩展。
  • 客户的特征：主动与服务器建立联系，客户之间通常不直接通信。客户的 IP 地址可以是动态的（比如移动设备）。
  • 典型应用：Web (HTTP)、电子邮件、传统文件传输。

核心概念: 对等体系结构 (Peer-to-Peer, P2P)

在该结构中，对位于数据中心的专用服务器的依赖极小甚至完全没有。应用程序主要利用边缘主机（即对等方 Peer）之间的直接通信来完成任务。

  • 自扩展性 (Self-scalability)：P2P 结构最引人注目的特性。虽然每个对等方都会通过请求文件产生工作负载，但它们也会通过向其他对等方分发文件来为系统增加服务能力。
  • 典型应用：BitTorrent (文件分发)、迅雷等。

易错辨析: C/S 与 P2P 的对比与辨析

在架构选择时需要权衡利弊：

  • C/S 架构：优势是易于管理、安全性高、中心化控制力强；劣势是服务器容易成为性能瓶颈和单点故障，且数据中心的建设和维护成本极高。
  • P2P 架构：优势是极佳的自扩展性，成本低；劣势是面临严峻的安全挑战，由于对等方主机频繁上下线带来的高度动态性，以及管理和计费极其困难。

进程通信 ​

在操作系统的层面上，真正在进行通信的实体并不是“主机”，而是运行在主机上的进程 (process)。

核心概念: 客户和服务器进程

对于每一对通信的进程，我们总是可以将其分为两端：

  • 客户进程 (Client process)：发起通信（即主动联系其他进程）的进程。
  • 服务器进程 (Server process)：等待联系（即被动等待接入）的进程。

注意：即使在 P2P 架构中，当一个对等方下载文件时它是客户，当它上传文件时它是服务器。

核心概念: 套接字 (Socket)

进程通过一个称为套接字的软件接口向网络发送报文和从网络接收报文。 可以将套接字比作一扇“门”： 进程（犹如在房子里的人）将报文推出这扇门；门外是网络基础设施。一旦报文推出了门，底层网络机制就负责将它传送到目的主机的门前；然后接收进程打开它的门，把报文拿进来。 套接字是应用层与运输层的接口，也是应用程序开发者能够控制网络交互的边界（也被称为应用程序编程接口，API）。

关键公式: 如何寻址一个目标进程？

为了向目标进程发送数据，源进程需要知道两个维度的信息：

  1. IP 地址 (IP Address)：用来在极其庞大的因特网中定位到哪一台主机。
  2. 端口号 (Port Number)：用来定位该主机上具体是哪一个进程（因为一台主机上可能同时运行着 Web、邮件等多个网络应用）。

常用的周知端口号（Well-known Port Numbers）举例：Web服务器 (HTTP) 为 80；邮件服务器 (SMTP) 为 25。

可供应用程序使用的运输服务 ​

不同的应用对网络传输的要求截然不同。运输层协议大致可以从以下四个维度为应用层提供服务：

核心概念: 运输服务的四大分类

  1. 可靠数据传输 (Reliable Data Transfer)：协议保证发送的数据字节流 100% 准确无误、按序地交付给接收方。不能容忍数据丢失的应用（如文件传输、金融交易）必须要求这种服务。
  2. 吞吐量 (Throughput)：协议能否保证在通信两端提供某种特定的、有保证的传输速率。具有严格带宽要求的应用被称为带宽敏感应用 (bandwidth-sensitive application)，如高清流媒体；而能适应不同可用带宽的应用称为弹性应用 (elastic application)，如电子邮件。
  3. 定时 (Timing)：协议能否保证发送端发出的比特在限定时间内到达接收端。这对实时交互应用（如互联网电话、多人对战游戏）至关重要。
  4. 安全性 (Security)：协议能否提供机密性（加密）、数据完整性以及端点认证。

因特网提供的运输服务 ​

因特网为应用层提供了两种极其基础但功能迥异的运输层协议：TCP 和 UDP。

易错辨析: TCP 与 UDP 服务模型的终极辨析

这是几乎所有计算机网络考试中最核心的考点：

  • TCP 服务：

    • 面向连接：在数据传输前，客户端和服务器必须互相握手建立连接。
    • 可靠数据传输：保证数据无差错、不丢失、不重复、按序到达。
    • 拥塞控制：当网络核心拥塞时，TCP 会主动“踩刹车”降低发送速率，这是一种全局的利他机制。
    • 缺点：开销大，建立连接有延迟，拥塞控制可能导致突发的速率降低。

  • UDP 服务：

    • 无连接：直接发送，不需要握手。
    • 不可靠数据传输：不保证数据能否到达，也不保证到达顺序。
    • 无拥塞控制：发送端可以尽可能快地将数据“倾倒”进网络，不管网络是否拥堵。
    • 优点：轻量，速度快，延迟极低。常用于实时语音视频通话。

注：因特网的 TCP 和 UDP 都不提供吞吐量和定时保证。

核心概念: 应用层的安全性：TLS

需要注意的是，TCP 和 UDP 本身提供的全都是明文传输，没有任何加密保护。为了解决这个问题，在应用层开发了 运输层安全性 (TLS, Transport Layer Security)（曾称为 SSL）。 TLS 不是一个新的独立运输层协议，而是对 TCP 的增强。它实现在应用层，提供加密、数据完整性和端点认证功能。

应用层协议 ​

核心概念: 应用层协议定义的内容

应用层协议 (Application-layer protocol) 定义了运行在不同端系统上的应用程序进程如何相互传递报文。具体来说，它规定了：

  • 交换的报文类型：如请求报文和响应报文。
  • 各种报文类型的语法：如报文中的各个字段及这些字段是如何描绘的。
  • 字段的语义：即这些字段中的信息的含义。
  • 规则：确定一个进程何时以及如何发送报文或对报文进行响应。

每天都有新的应用被开发出来。为了深入理解应用层，本章不会像百科全书一样罗列所有应用，而是挑选了5种最为重要和流行的网络应用进行详细探讨：

  • Web：极度流行的应用，其应用层协议HTTP相对简单且易于理解。
  • 电子邮件：因特网上第一个招人喜爱的应用程序，使用了多个应用层协议。
  • 目录服务 (DNS)：将易于记忆的主机名转换为网络地址，展示了核心网络功能如何在应用层实现。
  • P2P文件分发：展示了非中心化架构的强大扩展性。
  • 流式视频：包括经内容分发网 (CDN) 分发存储的视频流应用。

Web 和 HTTP ​

20世纪90年代初，万维网 (World Wide Web) 的出现极大地改变了人们的交流方式，它将因特网从少数研究人员的工具变成了大众不可或缺的基础设施。Web最吸引人的特点在于它的按需操作 (on-demand)，用户可以在需要时随时获取想要的内容。

HTTP 概述 ​

Web 的应用层协议是 超文本传输协议 (HTTP, HyperText Transfer Protocol)，它是 Web 的核心。HTTP 在两个程序中实现：一个客户程序（通常是 Web 浏览器）和一个服务器程序（即 Web 服务器）。

核心概念: Web 页面与对象

  • Web 页面 (Web page)：也叫文档，是由多个对象 (object) 组成的。
  • 对象：一个对象只是一个文件，如 HTML 文件、JPEG 图形、JavaScript 或视频片段，它们可通过 URL (统一资源定位符) 寻址。
  • HTML 基本文件 (base HTML file)：多数 Web 页面包含一个 HTML 基本文件以及几个引用对象。HTML 基本文件通过对象的 URL 来引用页面中的其他对象。

易错辨析: HTTP 对底层协议的依赖与无状态性

  • 底层协议：HTTP 严格要求可靠数据传输，因此它使用 TCP 作为它的支撑运输协议（而不是 UDP）。HTTP 客户首先发起一个与服务器的 TCP 连接，一旦连接建立，进程就可以通过套接字接口访问 TCP。
  • 无状态性：HTTP 服务器向客户发送被请求的文件，但不存储任何关于该客户的状态信息。如果一个客户在几秒内两次请求同一个对象，服务器会毫不犹豫地重新发送该对象，就像完全忘记不久前做过的事一样。因此，HTTP 被称为一个无状态协议 (stateless protocol)。

非持续连接和持续连接 ​

在许多网络应用中，客户和服务器会在一个较长的时间范围内通信，发出一系列请求。如果这一系列请求及其响应是经一个单独的 TCP 连接发送的，则称为 持续连接 (persistent connection)；如果每个请求/响应对都经一个单独的 TCP 连接发送，则称为 非持续连接 (non-persistent connection)。

核心概念: 非持续连接的 HTTP

在使用非持续连接时，每个 TCP 连接在服务器发送一个对象后就会关闭。这就意味着，如果一个 Web 页面包含 1 个 HTML 基础文件和 10 个 JPEG 图像，浏览器获取这个页面总共需要产生 11 个独立的 TCP 连接。 为了缩短响应时间，现代浏览器通常允许配置打开并行 TCP 连接来同时获取多个对象。

为了量化评估 Web 页面的请求时延，我们需要定义往返时间：

关键公式: 往返时间 (RTT, Round-Trip Time) 与总响应时间

RTT 是指一个短分组从客户发送到服务器，然后再返回客户所花费的总时间。它包含了分组传播时延、中间路由器的排队时延以及处理时延。

在使用非持续连接请求一个 HTML 文件的总时间计算中：

  • 第一个 RTT：客户发起 TCP 连接（三次握手的前两部分）。
  • 第二个 RTT：客户发送 HTTP 请求报文（结合三次握手的第三部分），服务器响应并返回 HTML 文件。

易错辨析: 非持续连接的缺点与持续连接的改进

  1. 非持续连接的缺点：首先，必须为每一个请求的对象建立和维护一个全新的 TCP 连接，这在客户和服务器上分配了大量的 TCP 缓冲区和状态变量，给 Web 服务器带来了严重的负担。其次，每一个对象都经受两倍 RTT 的交付时延。
  2. 持续连接 (Persistent Connection)：为了解决上述问题，HTTP/1.1 默认使用了持续连接。服务器在发送响应后保持 TCP 连接打开，后续对同一个服务器的请求和响应可以直接通过相同的连接传送，甚至可以通过流水线 (pipelining) 的方式连续发出请求，从而极大地避免了每次新建连接的巨大开销并降低了时延。

HTTP 报文格式 ​

HTTP 协议的规范包含了对两种 HTTP 报文的严格定义：请求报文 (request message) 和 响应报文 (response message)。

核心概念: HTTP 请求报文

一个典型的 HTTP 请求报文由以下几个部分组成：

  1. 请求行 (Request line)：报文的第一行。包含三个字段：
  • 方法字段：包括 GET, POST, HEAD, PUT, DELETE 等。
  • URL 字段：请求对象的路径。
  • HTTP 版本字段：如 HTTP/1.1。
  1. 首部行 (Header lines)：请求行之后的若干行，用于向服务器传递附加信息。常见的包括：\texttt{Host:} (指明对象所在的主机)、\texttt{Connection: close} (要求服务器发送完对象后关闭连接)、\texttt{User-agent:} (指明用户代理即浏览器的类型)、\texttt{Accept-language:} 等。
  2. 空行 (Blank line)：紧跟在首部行之后，标志着首部行的结束。
  3. 实体体 (Entity body)：使用 GET 方法时通常为空，而使用 POST 方法时（例如用户提交表单数据），该字段包含用户提交的具体内容。

易错辨析: GET 与 POST 方法的区别

虽然表单数据通常使用 POST 方法通过实体体发送，但 GET 方法也可以用来提交表单。当使用 GET 方法时，表单数据会被直接附加在被请求的 URL 之后（带有 \texttt{?} 和 \texttt{&} 等字符），这在传递少量非敏感数据时非常常见。 另外，HEAD 方法类似于 GET，但服务器在响应时不会返回请求对象，常用于网络调试和跟踪。

核心概念: HTTP 响应报文

HTTP 响应报文的结构与请求报文类似，包含：

  1. 状态行 (Status line)：包含协议版本、状态码 (Status code) 和相应的状态信息。
  2. 首部行 (Header lines)：如 \texttt{Connection: close}, \texttt{Date:} (服务器产生并发送该响应报文的日期和时间), \texttt{Server:}, \texttt{Last-Modified:} (对象最后修改时间), \texttt{Content-Length:} (被发送对象的字节数), \texttt{Content-Type:} 等。
  3. 空行。
  4. 实体体：包含了所请求的具体对象本身（如 HTML 代码或图片数据）。

关键公式: 常见的 HTTP 状态码及其含义

  • 200 OK：请求成功，信息在返回的响应报文中。
  • 301 Moved Permanently：请求的对象已经被永久转移，新的 URL 定义在响应报文的 \texttt{Location:} 首部行中。
  • 400 Bad Request：一个通用差错代码，指示该请求不能被服务器理解。
  • 404 Not Found：被请求的文档不在服务器上。
  • 505 HTTP Version Not Supported：服务器不支持请求报文使用的 HTTP 协议版本。

用户与服务器的交互：cookie ​

前面提到过，HTTP 是一个无状态的协议。然而，在很多场景下（如电子商务网站），服务器非常需要识别并限制用户访问，或者把用户的请求与该用户的身份联系起来。为了解决这个问题，HTTP 使用了 cookie 技术。

核心概念: Cookie 的四个核心组件

cookie 技术依赖于以下四个组件的配合：

  1. 在 HTTP 响应报文中包含一个 cookie 首部行（如 \texttt{Set-cookie: 1678}）。
  2. 在 HTTP 请求报文中包含一个 cookie 首部行（如 \texttt{Cookie: 1678}）。
  3. 在用户端系统中保留一个 cookie 文件，并由用户的浏览器进行管理。
  4. 在 Web 站点的后端数据库中保存关于这个 cookie 对应用户的详细信息。

易错辨析: Cookie 的双刃剑：状态与隐私

  • 好处：Cookie 允许网站创建用户会话状态 (user session state)，实现了诸如购物车、一键登录、个性化推荐等极大地提升用户体验的功能。
  • 坏处：Cookie 也引发了严重的隐私问题。许多网站（包括第三方广告网络）利用 cookie 在多个站点之间追踪用户的浏览行为，从而建立极为详细的用户画像。

Web 缓存 ​

Web 缓存器 (Web cache)，也称为 代理服务器 (proxy server)，它是代表初始 Web 服务器来满足 HTTP 请求的网络实体。

核心概念: Web 缓存的工作机制

Web 缓存器有自己的磁盘存储空间，并保存着最近请求过的对象的副本。 当用户通过浏览器配置指向某 Web 缓存器时：

  1. 浏览器首先向 Web 缓存器发送所有的 HTTP 请求。
  2. 如果缓存器中有该请求对象的副本，缓存器直接向客户浏览器返回包含该对象的 HTTP 响应报文。
  3. 如果缓存器中没有该副本，缓存器就会作为客户端，打开一个与初始源服务器的 TCP 连接，获取该对象，并在本地保存一份副本，然后将其发送给客户浏览器。

在此过程中，Web 缓存器既是服务器又是客户。

易错辨析: 部署 Web 缓存器的两大优势

  1. 它可以极大地降低客户端请求的响应时间（因为缓存器通常部署在离客户端更近的接入网中）。
  2. 它可以大幅度减少一个机构的接入链路到因特网的流量，从而整体上减轻了因特网核心的流量负担，这也是内容分发网络 (CDN) 的核心原理基础。

关键公式: 条件 GET (Conditional GET) 方法

虽然缓存技术很有用，但它引入了一个新问题：缓存器中保存的对象副本可能是陈旧的（初始服务器上的对象后来被修改了）。 HTTP 利用 条件 GET 机制来解决这个问题：

  • 请求报文使用 GET 方法。
  • 请求报文中包含 \hl{\texttt{If-Modified-Since:}} 首部行。该字段的值通常是缓存器上次获取该对象时的 \texttt{Last-Modified} 时间。
  • 初始服务器收到请求后进行检查：如果对象自该时间后没有被修改过，服务器将发送一个 304 Not Modified 的响应报文，并且不包含实体体（以节省带宽）。如果被修改过，则返回 \texttt{200 OK} 并附带最新的对象。

HTTP/2 ​

HTTP/1.1 虽然广泛使用了持续连接和流水线，但在面临包含海量小对象的复杂现代 Web 页面时，依然存在性能瓶颈。因此，HTTP/2 协议应运而生（随后的 HTTP/3 进一步优化了底层运输）。

核心概念: HTTP/2 的核心机制

HTTP/2 的主要目标是减少 Web 页面加载的时延：

  • 解决队头阻塞 (HOL blocking)：在 HTTP/1.1 的持续连接流水线中，如果最前面的请求遇到了处理瓶颈或是一个极大的视频对象，它会阻塞后面所有小对象的传输。HTTP/2 将 HTTP 报文划分为交错的帧 (frame)，可以在同一个 TCP 连接上并行且乱序地发送多个请求和响应，到达后再重新组装。
  • 首部压缩：由于 HTTP 请求和响应常常包含大量冗余的首部字段（如 Cookie），HTTP/2 引入了 HPACK 等算法来大幅压缩庞大的 HTTP 首部，降低带宽消耗。
  • 服务器推送 (Server Push)：服务器可以对客户端的一个特定请求主动发送多个响应，也就是主动将客户端稍后必然需要的相关对象提前“推送”到客户端缓存中。

因特网中的电子邮件 ​

电子邮件自因特网诞生初期就已经存在，并且时至今日依然是极其重要的应用。它是一种异步 (asynchronous) 通信媒介，即人们可以在方便的时候发送和阅读报文，而无须双方同时在线。

电子邮件系统的三大核心组件 ​

核心概念: 电子邮件系统的构成

一个高级别的因特网电子邮件系统主要由以下三个部分组成：

  1. 用户代理 (User Agent)：允许用户阅读、回复、转发、保存和撰写报文的软件（如微软的 Outlook、Apple Mail 或各种 Web 邮箱前端界面）。
  2. 邮件服务器 (Mail Server)：电子邮件体系结构的核心。每个接收方在某个邮件服务器上都有一个邮箱 (mailbox)，用于存放发送给他的报文。发送方则依靠服务器的报文队列 (message queue) 来暂存并尝试发送出去的邮件。
  3. 简单邮件传输协议 (SMTP)：这是因特网电子邮件中最重要的应用层协议。它负责将邮件从发送方的邮件服务器推送到接收方的邮件服务器。

易错辨析: SMTP 通信的端点辨析

初学者常常误以为 SMTP 是直接运行在两个用户终端（如两台电脑）之间的。实际上，SMTP 是两个邮件服务器之间的通信协议。 当 Alice 给 Bob 发邮件时，过程如下： Alice 的用户代理 Alice 的邮件服务器 Bob 的邮件服务器 Bob 的用户代理。

SMTP 协议机制 ​

SMTP 是因特网历史最悠久的协议之一（其核心 RFC 颁布于 1982 年）。它使用 TCP 作为底层可靠数据传输协议，周知端口号为 25。

核心概念: SMTP 的工作过程与特性

  • 直接传输：与很多网络协议不同，SMTP 通常不使用中间邮件服务器进行中继。如果 Alice 在北京，Bob 在纽约，TCP 连接是直接在这两个邮件服务器之间建立的。
  • 握手协议：在真正传输报文之前，SMTP 客户（发送方服务器）和 SMTP 服务器（接收方服务器）需要执行应用层握手。
  • 7 比特 ASCII 限制：这是 SMTP 一个非常古老且令人头疼的特性。它要求所有邮件报文的体部分（不仅仅是首部）必须是简单的 7 比特 ASCII 码。这意味着如果你要发送非英文字符、图片或附件，必须在发送前将其编码为 ASCII，并在接收后解码还原。

易错辨析: SMTP 与 HTTP 的深度对比

这两个协议都用于将文件从一台主机传送到另一台主机，但它们有着本质的区别：

  1. 推与拉：
  • HTTP 是一个 拉协议 (Pull Protocol)。用户使用 HTTP 从服务器上把文件“拉”下来，发起 TCP 连接的是想接收文件的机器。
  • SMTP 是一个 推协议 (Push Protocol)。发送方的邮件服务器主动把文件“推”向接收方的服务器，发起 TCP 连接的是想发送文件的机器。
  1. 数据格式：SMTP 严格限制使用 7 比特 ASCII，而 HTTP 数据不受此限制。
  2. 多对象处理：如果一个文档包含文字和 3 张图片，HTTP 会把每个对象封装在独立的响应报文中传送；而 SMTP 会把所有对象揉进同一个报文（使用 MIME 格式）中发送。

邮件访问协议 ​

既然 SMTP 是一个推协议，那么 Bob 的用户代理如何从他的邮件服务器上获取收到的邮件呢？Bob 显然不能使用 SMTP，因为拉取操作不符合 SMTP 的设计语义。因此，我们需要特殊的邮件访问协议 (Mail Access Protocol)。

核心概念: 三种主流邮件访问机制

  1. POP3 (邮局协议版本3)：一个极其简单的邮件访问协议。它的工作模式非常单一：用户代理连接上服务器，进行身份验证，然后下载邮件。它通常配置为“下载并删除”或“下载并保留”。POP3 在不同会话之间是无状态的。
  2. IMAP (因特网邮件访问协议)：比 POP3 复杂得多。它允许用户在服务器上创建多级文件夹并移动邮件。IMAP 服务器必须维护用户的状态信息。它非常适合现代用户在手机、平板和电脑多端同步邮件状态的需求。
  3. 基于 Web 的电子邮件 (HTTP)：如今最流行的访问方式（如 Gmail、QQ邮箱）。用户代理变成了普通的 Web 浏览器。用户无论发送邮件还是读取邮件，终端与邮件服务器之间的通信统统使用 HTTP，而只有各个邮件服务器之间依然使用传统的 SMTP。

DNS：因特网的目录服务 ​

因特网上的主机可以使用两种方式进行标识：一种是对人类友好的主机名 (hostname)（如 \texttt{www.google.com}），另一种是对路由器友好的定长、层次化的 IP 地址（如 \texttt{192.0.2.1}）。 我们需要一种能够进行主机名到 IP 地址转换的目录服务，这就是 域名系统 (DNS, Domain Name System)。

DNS 提供的服务 ​

DNS 运行在 UDP 之上，使用 53 号端口。它不仅提供最核心的 IP 转换服务，还提供一些其他重要功能。

核心概念: DNS 的主要功能

  • 主机名到 IP 地址的转换：这是最主要的功能，供 HTTP、SMTP 等其他应用层协议使用，以便在发起 TCP 连接前获取目标 IP。
  • 主机别名 (Host aliasing)：有着复杂规范主机名的主机可以拥有多个容易记忆的别名。DNS 可以解析别名，并给出规范主机名及其 IP。
  • 邮件服务器别名：使得公司的邮件地址（如 \texttt{bob@yahoo.com}）和 Web 地址（\texttt{www.yahoo.com}）可以使用相同的核心名字，而 DNS 能分辨出哪个指向 Web 服务器，哪个指向邮件服务器。
  • 负载分配 (Load distribution)：繁忙的站点通常在多台不同的服务器上运行（有不同的 IP）。DNS 数据库中可以为一个主机名存储一个 IP 地址集合。当客户端查询时，DNS 会将整个集合返回，但在每次回复时轮换 (rotate) 这些地址的顺序。因为客户端通常向排在最前面的 IP 发起连接，这就实现了将海量流量均匀分配到不同服务器上的目的。

DNS 的分布式层次数据库 ​

如果只有一个中心 DNS 服务器，它将面临单点故障、庞大通信容量、远距离时延和难以维护等致命问题。因此，DNS 采用了高度分布式的层次架构。

核心概念: DNS 服务器的层次结构

DNS 数据库被分布在遍及全球的各级服务器中：

  1. 根 DNS 服务器 (Root DNS server)：位于顶层，当本地服务器无法解析时，首先联系它。它提供负责各个顶级域的 TLD 服务器的 IP 地址。
  2. 顶级域 (TLD) DNS 服务器：负责顶级域名（如 \texttt{com}, \texttt{org}, \texttt{net}, \texttt{edu} 等）以及所有国家的顶级域名（如 \texttt{cn}, \texttt{uk}）。它们提供权威 DNS 服务器的 IP 地址。
  3. 权威 DNS 服务器 (Authoritative DNS server)：因特网上具有公共可访问主机的每个组织（如大学、公司）都必须提供公共的 DNS 记录，这些记录就存放在该组织的权威服务器中。

注：还有一种本地 DNS 服务器 (Local DNS server)，它严格来说不属于该层次结构，但它至关重要。每个 ISP 都有一台本地 DNS，它充当代理 (proxy) 的角色，代表本地主机向 DNS 层次结构发出查询请求。

易错辨析: 递归查询与迭代查询机制对比

当本地 DNS 服务器无法直接响应主机的查询时，有两条不同的寻址路径：

  • 递归查询 (Recursive query)：查询像接力棒一样传递。“我不知道，我帮你问上面，上面再去问上面，最后把答案一路传回来”。这种方式会将沉重的查询负担全部甩给层次较高的服务器（尤其是根服务器），因此在核心网络中极少使用。
  • 迭代查询 (Iterative query)：大部分 DNS 查询采用的方式。“我不知道这个域名的 IP，但我知道谁可能知道，你自己去问他”。例如：本地服务器问根服务器，根服务器返回 TLD 服务器的 IP；本地服务器再去问 TLD 服务器，TLD 返回权威服务器的 IP；本地服务器最后问权威服务器获取最终 IP。

在实际应用中，通常从“请求主机”到“本地 DNS 服务器”的这一步是递归的，而其余所有查询都是迭代的。

DNS 缓存与记录 ​

核心概念: DNS 缓存 (DNS Caching)

为了改善时延并减少在因特网上到处传输的 DNS 报文数量，DNS 广泛使用了缓存技术。 当一个 DNS 服务器接收到一个 DNS 回答时，它能将映射缓存在本地内存中。如果在一段时间内有另一台主机发起相同的查询，该 DNS 服务器就能直接提供缓存的 IP 地址，而无须再向其他服务器发送查询。为了防止数据过时，缓存记录会在一段时间（通常为两天）后被丢弃。

BitTorrent ​

BitTorrent 是一种用于文件分发的极其流行的 P2P 协议。

核心概念: BitTorrent 的核心术语与机制

  • 洪流 (Torrent)：参与一个特定文件分发的所有对等方的集合。
  • 块 (Chunk)：文件被划分为固定大小的数据块（通常为 256KB）。对等方在洪流中下载和上传的就是这些独立的块。
  • 追踪器 (Tracker)：每个洪流都有一个基础设施节点。当一个对等方加入洪流时，它向追踪器注册自己，并周期性地通知追踪器它仍在洪流中。追踪器负责给新加入的节点随机提供一个其他对等方的 IP 列表。

当一个对等方（比如 Alice）加入网络并获取到邻居列表后，她需要决定两件事：从邻居那里请求哪些块？以及将自己的上传带宽分配给哪些邻居？

易错辨析: 最稀缺优先 vs. 一报还一报 (Tit-for-tat)

这两个算法构成了 BitTorrent 高效运作的灵魂，绝不能混淆：

  1. 最稀缺优先 (Rarest First)：这是请求策略。Alice 会向邻居询问他们拥有哪些块，然后优先请求那些在邻居中最罕见的块。这样可以迅速增加系统中稀缺块的副本数量，均衡整个洪流中的块分布，并降低某个罕见块完全丢失的风险。
  2. 一报还一报 (Tit-for-tat)：这是响应策略（决定把数据发给谁）。为了防止某些节点“白嫖”（只下载不上传），Alice 会持续测量所有向她提供数据的邻居的下载速率，并挑选出向她发送数据最快的 4 个邻居。Alice 只会把自己的上传带宽分配给这 4 个节点（这称为将它们 疏通 unchoked），其他节点则被阻塞 (choked)。此外，Alice 每隔 30 秒会随机挑选一个额外的邻居进行疏通（称为 试探性疏通 optimistic unchoking），以便发现可能提供更高下载速率的新伙伴。

视频流和内容分发网 ​

自 2000 年代后期以来，因特网最引人注目的变化莫过于流式视频的爆发式增长（如 YouTube、Netflix）。视频流量占据了如今因特网下行流量的绝大部分。为了将海量的高清视频顺畅地分发给全球用户，内容分发网 (CDN) 技术应运而生。

因特网视频 ​

与传统的文本和图像不同，视频是一种高密度的数据，且对网络传输的平均吞吐量有着极高的要求。

核心概念: HTTP 流和 DASH

早期的流式视频直接通过标准的 HTTP GET 请求下载，这种方式缺乏适应性，容易在网络拥塞时导致视频卡顿。 现代的流式视频大多采用了 经 HTTP 的动态适应性流 (DASH, Dynamic Adaptive Streaming over HTTP) 技术。

  • 服务器端：视频被编码为几个不同的版本（例如 4K, 1080p, 720p），每个版本对应不同的比特率，并被分割成几秒长的小片段。服务器还会提供一个 告示文件 (manifest file)，列出所有版本及其 URL。
  • 客户端：客户端首先请求告示文件，然后在播放过程中，不断地测量自身与服务器之间的可用接收带宽。客户端会动态地决定下一个片段请求哪个质量的版本。如果带宽充裕，请求高画质；如果带宽恶化，迅速切换到低画质以保证播放不中断。

结论：在 DASH 中，是客户端智能地决定在何时、以何种速率请求何种块，而不是由服务器盲目推送。

内容分发网 (CDN) ​

如果所有的视频都存放在位于某个国家的单一超大巨型数据中心，那么远在地球另一端的用户在观看视频时，必将面临长达数百毫秒的端到端时延、无数中间链路的瓶颈拥塞以及高昂的通信成本。

核心概念: CDN 的部署策略

CDN (Content Distribution Network) 通过在地理上分散的服务器集群中复制内容来解决这个问题。CDN 主要有两种服务器安置策略：

  1. 深入 (Enter Deep)：如 Akamai 采用的策略。将服务器集群尽可能深地部署在全世界大量的接入 ISP 中。目标是极其靠近端用户，最大限度减少源端到端系统之间的链路数量和时延。缺点是集群数量极为庞大，管理和维护成本极高。
  2. 邀请做客 (Bring Home)：如 Limelight 采用的策略。在较少量的关键位置（如大型的因特网交换点 IXP）建设包含庞大集群的超级节点。这种方式没有那么靠近终端用户，但极大降低了维护和管理的复杂度与成本。

例题: CDN 与 DNS 拦截技术的配合

当用户点击一个视频链接时，视频网站如何神不知鬼不觉地将用户的请求重定向到离他最近的 CDN 服务器上？这完全归功于 DNS 的拦截与重定向机制。

  1. 用户的本地 DNS 服务器请求解析视频的主机名（如 \texttt{video.netcinema.com}）。
  2. 视频网站的权威 DNS 服务器不返回具体的 IP 地址，而是返回一个属于 CDN 供应商的主机名（如 \texttt{a1105.cdn.com}）。这就将 DNS 请求交接给了 CDN 的体系。
  3. 本地 DNS 服务器继而向该 CDN 的权威 DNS 服务器发起查询。
  4. CDN 的 DNS 服务器会根据用户的本地 DNS IP地址，计算出物理距离或网络拓扑上最优的（通常也是最快的）CDN 边缘服务器的 IP 地址，并将其返回。
  5. 用户最终与这台被选定的 CDN 服务器建立 TCP 连接并获取流式视频。

套接字编程：生成网络应用 ​

在前面的小节中，我们学习了网络应用是如何在各种协议的指导下工作的。本节我们将视线转移到开发者的角度，探讨如何利用 套接字 (Socket) API 编写能在网络中通信的客户端和服务器程序。

UDP 套接字编程 ​

运行在不同机器上的进程通过向套接字发送报文来进行通信。使用 UDP 时，发送进程必须将目的地的地址（IP地址和端口号）附着在每一个分组上。

核心概念: UDP 套接字工作流程

  1. 客户侧：客户端创建一个 UDP 套接字，将应用层数据加上目的 IP 和端口号，封装成数据报，然后将其推入套接字。
  2. 网络传输：因特网的底层路由器使用数据报中的目的 IP 地址将分组路由到目的主机。
  3. 服务器侧：服务器必须有一个运行的进程和绑定的套接字来监听特定端口。当数据报到达时，服务器从套接字中提取出数据，以及源主机的 IP 和端口号（以便服务器稍后能够向客户端发送响应）。

易错辨析: UDP 通信的“无状态”与不可靠性

在 UDP 套接字编程中，客户端和服务器之间不存在握手，也没有建立任何持久的连接（管道）。每个发送的数据报都是一个独立的实体。如果网络拥塞，数据报可能会在传输途中丢失，或者乱序到达接收方。UDP 协议及底层的 API 绝不会通知应用程序数据是否安全抵达。因此，所有关于重传和纠错的逻辑必须由应用层开发者自己实现。

TCP 套接字编程 ​

与 UDP 不同，TCP 是一种面向连接的协议。在互相发送数据之前，客户端和服务器必须先建立一条可靠的 TCP 连接。

核心概念: TCP 套接字工作流程

  1. 服务器准备：服务器进程必须先运行，并创建一个 欢迎套接字 (Welcoming Socket)，绑定在某个周知的端口上，专门用于被动等待客户端的连接请求。
  2. 客户端发起连接：客户端创建一个 TCP 套接字，并指定服务器的 IP 和端口号，主动发起建立连接的 三次握手。
  3. 服务器生成新套接字：当服务器的欢迎套接字收到连接请求时，它会为这个特定的客户“派生”出一个全新的 连接套接字 (Connection Socket)。
  4. 数据传输：此后，这对客户和服务器进程就通过这个专用的连接套接字，以字节流 (Byte Stream) 的形式可靠地、按序地交换数据。

易错辨析: 欢迎套接字与连接套接字的终极辨析

初学者常对 TCP 服务器中为什么需要两种不同的套接字感到困惑：

  • 欢迎套接字：就像是餐厅门口的迎宾员。它永远固定在那个门口（周知端口），任务只有一个——接待新来的客人，但不负责具体的上菜和服务。
  • 连接套接字：就像是餐厅里专门为你这桌服务的专属服务员。迎宾员把你领进门后，会指派一个专门的服务员（连接套接字）负责你后续的所有点餐和沟通需求。有了这种机制，一个 TCP 服务器就能并发地同时服务成千上万个客户端，而绝对不会将它们的数据混淆。

例题: 基于 TCP 的套接字编程实战特性

在实战编程（如使用 Python、Java 或 C）中，一旦客户端的 \texttt{connect()} 函数成功返回，意味着 TCP 的三次握手已经完成。此时，开发者只需要调用简单的 \texttt{send()} 和 \texttt{receive()} 函数即可传输数据，完全不需要再像 UDP 那样在每次发送数据时手动指定服务器的 IP 和端口号，因为底层的 TCP 状态机已经默默记录了这条逻辑连接的所有端点信息。

运输层 ​

在应用层，我们看到了应用程序是如何利用网络进行通信的。现在我们将沿着协议栈向下深入一层，来到运输层 (Transport Layer)，探究它是如何为运行在不同主机上的应用进程提供极其重要的端到端通信服务的。

运输层服务和概述 ​

运输层协议为运行在不同端系统上的应用进程之间提供了逻辑通信 (Logical Communication) 功能。这意味着，从应用程序的角度来看，运行着进程的主机仿佛是直接相连的物理线缆，哪怕它们实际上相隔了整个地球和无数个可能拥塞的路由器。

运输层和网络层的关系 ​

在协议栈中，运输层紧紧位于网络层之上。深刻理解这两层的区别和功能边界，是掌握计算机网络原理的重中之重。

核心概念: 进程间通信与主机间通信

  • 网络层：提供了主机 (Host) 到主机之间的逻辑通信。它的任务是将分组（网络层数据报）从源物理机器送达到目的物理机器。
  • 运输层：提供了进程 (Process) 到进程之间的逻辑通信。它利用网络层的服务，进一步将数据准确无误地投递给目的主机上成百上千个正在运行的应用程序中的某一个。

易错辨析: 经典类比：家庭信件的收发机制辨析

为了彻底弄清运输层和网络层的关系，原书提供了一个极其精妙的类比： 假设有两家人，住在东海岸的 Ann 家和西海岸的 Bill 家，每家都有 12 个堂兄弟姐妹，他们互相之间每周频繁写信。

  • 应用层报文 = 信封里的信件内容。
  • 进程 = 孩子们（真正写信和读信的实体）。
  • 主机 (端系统) = Ann 家和 Bill 家的物理房子。
  • 运输层协议 = Ann 和 Bill（家庭内部的信件总管代表）。Ann 负责把家里弟弟妹妹写好的信收集起来交给邮递员；并在收到邮递员送来的信后，根据信封上的名字把信精准分发给具体的弟弟妹妹。
  • 网络层协议 = 邮政服务（邮递员）。负责把包含所有信件的邮包从东海岸的房子物理搬运到西海岸的房子门前，但他绝不负责进屋分发。

在这个类比中，Ann (运输层) 的工作质量完全受限于邮政系统 (网络层) 的能力：如果邮局承诺信件一定会到，Ann 就很轻松；如果邮局经常丢信（就如同因特网的 IP 协议不可靠一样），Ann 如果想向弟弟妹妹保证信件绝对不丢，她就必须自己建立一套“记账和要求对方重寄”的复杂协议系统（这正是 TCP 协议诞生的原因！）。

因特网运输层概述 ​

因特网（即 TCP/IP 网络）的核心向应用层提供了两种截然不同哲学的运输层协议。

核心概念: UDP 与 TCP 的基本职责与扩充

我们在上一章已经简要对比过这两种协议。在运输层，它们的首要任务是将底层的 IP 主机交付服务，扩展为进程级的交付服务，这个关键过程被称为多路复用与多路分解 (Multiplexing and Demultiplexing)。

  • UDP (用户数据报协议)：它被设计得极其精简。除了提供多路复用/多路分解功能以及极其基础的差错检测（校验和）之外，它几乎没有给 IP 服务增加任何实质性的东西。它是一种不可靠的、无连接的尽力而为的服务。
  • TCP (传输控制协议)：它极其复杂。它提供了强悍的可靠数据传输 (Reliable Data Transfer) 能力。通过序号、确认、定时器和重传机制，TCP 能将底下漏风、丢包的 IP 网络转变为一条可靠的数据传输管道。此外，它还提供利他的拥塞控制 (Congestion Control)。

多路复用与多路分解 ​

为了让底层的网络层（只负责主机到主机的交付）能够为应用层的多个进程提供服务，运输层必须具备一种分发机制。这就是多路复用 (Multiplexing) 与多路分解 (Demultiplexing)。

核心概念: 多路复用与多路分解的严格定义

  • 多路分解（接收端）：将运输层报文段中的数据，交付到目的主机上正确的套接字 (Socket) 的工作。
  • 多路复用（发送端）：在源主机从不同套接字中收集数据块，并为每个数据块封装上首部信息（这将在以后用于分解）从而生成报文段，然后将这些报文段传递到网络层的工作。

为了实现多路复用和多路分解，运输层报文段的首部必须包含特殊字段来指示该报文段究竟属于哪个套接字。这两个极度重要的字段就是：源端口号字段 (Source port number field) 和 目的端口号字段 (Destination port number field)。端口号是一个 16 比特的整数，大小在 之间。其中 被称为周知端口号，保留给严格受控的应用层协议。

易错辨析: 无连接 (UDP) 与面向连接 (TCP) 的多路分解机制辨析

这是运输层最常考的底层机制差异，决定了服务器如何区分海量的并发请求：

  1. UDP 的多路分解（二元组标识）： 一个 UDP 套接字是由一个二元组来全面标识的，即 (目的 IP 地址, 目的端口号)。 这意味着，如果两个 UDP 报文段具有不同的源 IP 地址或源端口号，但具有相同的目的 IP 和目的端口号，那么这两个报文段将通过同一个目的套接字，被定向到同一个进程。
  2. TCP 的多路分解（四元组标识）： 一个 TCP 连接套接字是由一个四元组来标识的，即 (源 IP 地址, 源端口号, 目的 IP 地址, 目的端口号)。 当一个 TCP 报文段到达时，接收主机将使用全部 4 个值来将报文段定向到相应的套接字。这意味着，即使两个报文段的目的 IP 和目的端口完全相同，只要它们的源 IP 或源端口有任何一个不同，它们也会被定向到服务器上两个完全不同的套接字（除了携带连接请求的初始 SYN 报文段会去往欢迎套接字外）。

无连接运输：UDP ​

UDP (用户数据报协议) 是一个轻量级的运输协议，它几乎没有为底层的 IP 添加任何多余的东西，被称为是一种**“无修饰的” (no frills)**、“极其简单的”协议。

为什么有些应用偏爱 UDP ​

既然 TCP 提供了绝对可靠的数据传输服务，为什么还要设计一个不保证数据到达的 UDP 呢？因为在很多特殊的应用场景中，UDP 有着不可替代的优势。

核心概念: 选择 UDP 而非 TCP 的四大核心理由

  1. 关于何时、发送什么数据的应用层控制更为精细：TCP 有严格的拥塞控制，当网络拥塞时会强制降低发送速率，并不断重传直到成功（这可能导致严重的延迟）。而 UDP 会把应用层交下来的数据毫不迟疑地立刻打包发送，非常适合容忍一定丢包但绝不能容忍延迟的实时交互应用（如微信语音、实时在线游戏）。
  2. 无须建立连接：TCP 需要三次握手，引入了巨大的初始时延。UDP “拔枪就射”，没有任何预备动作（这正是 DNS 选择 UDP 而非 TCP 的主要原因）。
  3. 无连接状态：TCP 需要在端系统中维护复杂的接收/发送缓存、拥塞参数以及序号确认号。UDP 不维护任何状态，因此一台高配服务器使用 UDP 能比使用 TCP 支持多得多的活跃客户。
  4. 分组首部开销小：每个 TCP 报文段有长达 20 字节的固定首部开销，而 UDP 只有可怜的 8 字节。

UDP 报文段结构 ​

UDP 报文段的结构极度精简，其首部只有 4 个字段，每个字段 2 个字节，总共 8 个字节。

核心概念: UDP 报文段首部字段

  • 源端口号 (Source port) (16 bits)
  • 目的端口号 (Destination port) (16 bits)
  • 长度 (Length) (16 bits)：指示了在 UDP 报文段中的字节数（首部加上数据）。
  • 检验和 (Checksum) (16 bits)：用于接收方检查该报文段在传输过程中是否出现了比特差错。

UDP 检验和 ​

UDP 检验和提供了差错检测功能。它用于确定当 UDP 报文段从源到达目的地移动时，其中的比特是否发生了改变（例如，由于链路中的物理噪声或路由器的存储器错误）。

关键公式: UDP 检验和的计算原理

发送方的操作：

  1. 将报文段中的所有 16 比特字相加（如果有进位，则将进位加到最低位上，称为回卷 end-around carry）。
  2. 将相加得到的结果进行反码 (1s complement) 运算（即 0 变 1，1 变 0）。
  3. 将这个反码结果放入 UDP 首部的检验和字段。

接收方的操作： 接收方将收到的报文段（包括数据、首部，并将检验和字段也加进去）所有的 16 比特字相加。如果传输完全没有差错，那么最终的加法结果必然是全 1（即 1111111111111111）。如果其中有任何一个 0，就说明在传输中出现了差错。

易错辨析: 端到端原则 (End-to-End Principle) 的深刻体现

你可能会问：既然许多链路层协议（如以太网）已经提供了强大的 CRC 差错检测机制，为什么 UDP 还要在运输层再做一遍检验和？

  • 首先，不能保证源和目的之间的所有链路都提供差错检测（也许中间有一段极其古老的串行链路不提供校验）。
  • 其次，即使链路完全无差错，当报文段存储在某台路由器的内存中排队等待转发时，也极有可能因为硬件故障被随机翻转比特。
  • 结论：由于较低层协议无法提供绝对的可靠性保障，因此必须在端到端的基础上在较高层实现差错检测功能。这就是网络系统设计中极为经典的端到端原则。

注：UDP 虽然提供差错检测，但它不提供差错恢复。如果查出错误，UDP 通常只会简单地丢弃该受损的报文段，并不会通知发送方重传。

可靠数据传输原理 ​

可靠数据传输 (Reliable Data Transfer, RDT) 不仅在运输层，在链路层和应用层也是一个极其核心且经典的计算机网络问题。其基本前提是：底层的信道可能是不可靠的（会发生比特翻转或丢包），我们需要在不可靠的信道之上，通过精妙的协议设计，向最高层抽象出一个绝对可靠的逻辑信道。

构造可靠数据传输协议 ​

我们将通过构建一系列状态机，逐步设计一个完整的可靠数据传输协议，从最简单的理想模型开始，逐步加入对现实网络中各种错误的应对机制。

核心概念: 从 rdt 1.0 到 rdt 3.0 的演进

  • rdt 1.0：经完全可靠信道的可靠数据传输。假设底层信道绝对不丢包、不翻转比特。发送方直接发，接收方直接收，不需要任何反馈。

  • rdt 2.0：经具有比特差错信道的可靠数据传输。假设分组不会丢失，但里面的比特可能会因为物理噪声而翻转。引入了 自动重传请求 (ARQ) 协议的三大核心机制：

    • 差错检测：如检验和 (Checksum)，用于接收方判断分组是否出错。
    • 接收方反馈：肯定确认 (ACK) 和否定确认 (NAK)。接收方借此告诉发送方分组是否完好。
    • 重传：发送方收到 NAK 时，主动重传该出错的分组。

  • rdt 2.1 与 2.2：rdt 2.0 有一个致命的盲区——ACK/NAK 分组本身在传输中也可能受损！如果发送方收到一个乱码的 ACK，它不知道到底该不该重传。为了解决这个问题，我们在 rdt 2.1 中为数据分组引入了 序号 (Sequence Number)（对于停等协议，只需要 0 和 1 交替即可区分新分组和重传分组）。在 rdt 2.2 中，我们进一步简化，去掉了 NAK，仅使用带有序号的 ACK（如果发送方收到对上一个序号的重复 ACK，就相当于收到了当前分组的 NAK）。

  • rdt 3.0：经具有比特差错和丢包信道的可靠数据传输。现实网络不仅会错，还会把分组直接丢弃。为了应对丢包（无论是数据包丢了还是 ACK 丢了），引入了 倒数定时器 (Countdown Timer)。如果发送方在规定的时间内没有收到任何回复，就直接盲目重传该分组。

易错辨析: 停等协议 (Stop-and-Wait) 的本质局限

rdt 3.0 是一个在功能上完全正确的协议，但它是一个极其低效的停等协议。发送方发送一个分组后，必须停下来等待确认，才能发送下一个分组。在长距离、高带宽的现代物理链路（如跨洋光缆）中，这种机制会导致物理链路的实际利用率低得令人发指！

回退 N 步 (GBN) 与选择重传 (SR) ​

易错辨析: GBN 与 SR 的终极对比与辨析

在应对由于流水线引发的丢包和乱序时，GBN 和 SR 采取了截然不同的处理哲学，这是各大网络考试中的极高频考点：

  1. 回退 N 步 (Go-Back-N, GBN)：
  • 累积确认 (Cumulative ACK)：接收方发送的 ACK 序号 具有强烈的包容性，它表示：序号 及之前的所有分组都已按序、正确地被接收。
  • 直接丢弃失序分组：接收方绝不缓存失序的分组。如果期望收到 2 号包，却提前收到了 3 号包，接收方会直接丢弃 3 号包，并重新发送对 1 号包的 ACK。
  • 单一重传机制与定时器：发送方只为最古老的未确认分组维护一个单一的定时器。如果该定时器超时，发送方必须重传所有已发送但还未被确认的分组（即“回退 N 步”）。
  1. 选择重传 (Selective Repeat, SR)：
  • 独立确认：接收方对每个正确接收的分组单独发送 ACK，不具有累积性。
  • 失序缓存：接收方会缓存失序到达的健康分组，直到所有缺失的报文段被后续重传填补，才将它们按序组装并一起交付给上层应用。
  • 精准重传与独立定时器：发送方为每一个发送的分组维护一个独立的定时器。当某个超时发生时，只精准重传那一个丢失的分组。

结论：当网络信道误码率或丢包率较高时，GBN 的“连坐”机制会导致大量已经被正确接收的分组被不必要地重传，严重浪费带宽；而 SR 虽然能避免无意义的重传，但其双方的缓存管理和定时器维护开销要庞杂得多。

面向连接的运输：TCP ​

前面我们学习了为了实现可靠数据传输所采用的一系列通用原理，现在我们将探索因特网的运输层主力协议——传输控制协议 (TCP) 是如何在真实的不可靠 IP 网络上实现这些原理的。

TCP 连接概述 ​

核心概念: TCP 连接的核心特征

  • 面向连接的 (Connection-oriented)：在两个进程可以开始发送应用层数据之前，它们必须先相互“握手”来初始化与许多TCP状态变量相关的连接参数。
  • 全双工服务 (Full-duplex service)：如果主机A和主机B之间存在一条TCP连接，那么应用层数据就可以在从A流向B的同时，也从B流向A。
  • 点对点 (Point-to-point)：TCP 连接始终是单个发送方和单个接收方之间的逻辑连接。TCP 绝对不支持从一个发送方向多个接收方广播数据的“多播”服务。
  • 发送缓存 (Send buffer) 和 MSS：TCP将应用层传下来的数据引导到发送缓存中。TCP每次从缓存中抓取并放置到一个报文段里的数据量受到最大报文段长度 (MSS, Maximum Segment Size) 的限制。MSS通常受限于链路层的最大传输单元 (MTU)（典型以太网MTU为1500字节，减去TCP/IP首部的40字节，MSS的典型值为1460字节）。

TCP 报文段结构 ​

核心概念: TCP 首部的关键字段

TCP报文段由首部和数据字段组成。其首部通常为 20 字节（不含可变长的选项字段）。核心字段包括：

  • 源端口号和目的端口号（各16比特）：用于多路复用与多路分解。
  • 序号 (Sequence number)（32比特）和 确认号 (Acknowledgment number)（32比特）：TCP实现可靠数据传输的基石。
  • 接收窗口 (Receive window)（16比特）：用于流量控制，指示接收方愿意接受的剩余缓冲字节数量。
  • 首部长度 (Header length)（4比特）：因为TCP选项字段变长，所以需要指示首部真正结束、数据开始的位置。
  • 标志字段 (Flag field)（6比特）：
    • ACK：指示确认号字段包含的值是有效的。
    • SYN / FIN：分别用于连接的建立和拆除。
    • RST：用于重置连接。
    • PSH / URG：用于推送数据和紧急数据指针（现代应用实践中已极少使用）。

易错辨析: TCP 的序号：基于字节流的视角陷阱

这是 TCP 协议中最容易混淆、也是考试中最常设坑的地方：TCP 的序号和确认号是建立在传输的字节流之上，而绝对不是建立在传送的报文段的序列之上！

  • 序号的含义：该报文段首字节的字节流编号。假设数据流为500,000字节，MSS为1000字节。第一个报文段的序号为0，第二个报文段的序号就是 1000（而不是1！），第三个就是 2000。
  • 确认号的含义：主机期望从对方收到的下一个字节的序号。如果主机A收到了B发来的编号为0到535的字节流，A发送给B的确认号将是 536。

注：由于TCP提供累积确认 (Cumulative ACK)，所以当A向B发出确认号536时，它也同时向B做出了严正保证：“535及之前的所有字节我都已经完美无缺地收到了”。

可靠数据传输机制的演进 ​

TCP 在 IP 的不可靠服务之上创建了高度可靠的数据传输服务。其核心机制巧妙地结合了 GBN（累积确认）和 SR（缓存失序分组）的优点。

核心概念: TCP 重传的三大触发与优化机制

  1. 单一重传定时器：TCP 发送方即使有多个未确认的报文段在飞行，概念上也只维护一个重传定时器（通常与最老的未确认报文段关联）。
  2. 超时加倍 (Doubling the Timeout Interval)：每次发生超时重传时，TCP 不再通过 EstimatedRTT 重新计算，而是直接将下一次的超时时间间隔强制加倍。这是一种初级的拥塞控制机制，目的是防止在网络已经严重拥塞丢包时，发送方还持续猛烈地以短周期重传，从而导致网络彻底崩溃。
  3. 快速重传 (Fast Retransmit)：超时触发重传的问题在于超时周期通常很长，极大地拖慢了恢复速度。如果接收方收到一个失序的报文段，它会立即发送对最后一个按序字节的冗余ACK。如果发送方连续收到3个冗余 ACK（即对同一个字节的4次确认），它会认为该确认号对应的报文段已经在网络中丢失，并立即在定时器过期之前重传该报文段！

流量控制 ​

易错辨析: 流量控制 vs. 拥塞控制 的终极辨析

这两个概念经常被混为一谈，但它们的初衷和发生位置截然不同：

  • 流量控制 (Flow Control)：这是一个局部、端到端的问题。如果发送方发送得太快，接收方应用进程提取数据的速度跟不上，接收方的接收缓存 (Receive Buffer) 就会被填满甚至溢出。TCP 提供流量控制纯粹是为了消除接收方缓存溢出的可能性。
  • 拥塞控制 (Congestion Control)：这是一个全局问题。由于 IP 网络核心的路由器排队队列可能因海量并发流量而溢出丢包。TCP 主动降低发送速率是为了防止网络基础设施本身崩溃。

TCP 是如何实现流量控制的？ 接收方通过在其发送的每一个 TCP 首部中填写 接收窗口 (rwnd) 字段，明确告诉发送方自己缓存中还有多少可用空间。发送方会严格保证其“未确认数据量”绝对不超过 rwnd 的值。

TCP 连接管理 ​

建立和拆除连接是 TCP 协议中最著名、最经典的流程。

核心概念: 三次握手 (Three-way Handshake) 建立连接

  1. 第一步 (SYN)：客户端主动发起，向服务器发送一个特殊的 TCP 报文段（不含应用层数据）。其首部中的 SYN 比特置为 1，并随机选择一个初始序号 (client_isn) 填入序号字段。
  2. 第二步 (SYNACK)：服务器收到 SYN 报文段后，为该 TCP 连接分配所需的 TCP 缓存和状态变量。服务器发回一个允许连接的响应：其 SYN 比特也置为 1，确认号填为 client_isn + 1，并随机选择自己的初始序号 (server_isn)。这个特殊的报文段被称为 SYNACK 报文段。
  3. 第三步 (ACK)：客户端收到 SYNACK 后，也正式为连接分配缓存和变量。向服务器发送最后一个确认，此时 SYN 比特必须置为 0，确认号为 server_isn + 1。从这步开始，报文段的实体体中可以携带实际的应用层数据！

核心概念: 四次挥手 (Four-way Teardown) 拆除连接

以客户端主动关闭连接为例：

  1. 客户端应用进程发出关闭指令，TCP 客户端发送一个首部中 FIN 比特置为 1 的控制报文段，进入 \texttt{FIN_WAIT_1} 状态。
  2. 服务器收到 FIN 报文段，返回一个确认 ACK，进入 \texttt{CLOSE_WAIT} 状态。此时服务器如果还有数据没传完，可以继续传。客户端收到 ACK 后进入 \texttt{FIN_WAIT_2} 状态。
  3. 服务器数据全部发送完毕后，也发送一个 FIN 比特置为 1 的报文段，进入 \texttt{LAST_ACK} 状态。
  4. 客户端收到服务器的 FIN，返回最终的 ACK，进入 \hl{\texttt{TIME_WAIT}} 状态。

\textit{关键细节：客户端在 \texttt{TIME_WAIT} 状态通常会强制等待一段时间（如 30 秒或 1 分钟）。这是为了防止最后发出的 ACK 丢失导致服务器不停重传 FIN，同时也是为了确保网络中迟到的、属于该旧连接的残余报文段全部消失，避免污染未来的新连接。}

拥塞控制原理 ​

在实践中，丢包一般是当网络变得拥塞时由于路由器缓存溢出引起的。因此，我们在上一节中学习的重传机制（可靠数据传输），虽然能作为丢包的对策，但如果任由发送方无节制地重传，只会让拥塞的网络雪上加霜。

拥塞的原因与代价 ​

核心概念: 拥塞的代价

当网络面临拥塞（即系统中承载的流量接近或超过物理链路极限）时，会产生严重的代价：

  • 极大的排队时延：当分组的到达速率接近链路容量时，路由器缓存中的队伍会变得极长，导致网络延迟飙升。
  • 无用的重传（浪费容量）：发送方必须执行重传以补偿因为缓存溢出而丢弃的分组。更糟的是，由于排队时延过大引发定时器超时，发送方可能会重传那些并未丢失但在队列中苦苦等待的分组，这导致网络链路的带宽被同一分组的多个副本白白消耗。
  • 上游路由器资源的浪费：当一个分组在经过多跳路由器后，如果在最后一跳因为拥塞被丢弃，那么该分组在前面所有路由器上消耗的传输容量和排队空间都将被彻底浪费。

拥塞控制方法 ​

易错辨析: 端到端拥塞控制 vs. 网络辅助的拥塞控制

在宏观的网络架构设计上，解决拥塞通常有两种截然不同的方法：

  1. 端到端拥塞控制 (End-to-end congestion control)：网络层（路由器）不向端系统提供任何显式的拥塞反馈信息。端系统必须通过观察网络行为（如分组丢失、往返时延的剧增）来自行推断网络是否拥塞。这是经典 TCP 采用的方法。
  2. 网络辅助的拥塞控制 (Network-assisted congestion control)：网络内部的路由器主动向发送方提供显式的反馈状态（例如，通过设置分组首部中的某个拥塞指示比特，如 ECN 比特），明确告诉发送方“我已经拥塞了，请减速”。

TCP 拥塞控制 ​

由于早期的 IP 层不向端系统提供任何反馈，TCP 必须使用端到端的拥塞控制。

TCP 拥塞控制算法 ​

TCP 拥塞控制算法 (TCP Congestion Control Algorithm) 包含三个主要的状态（阶段），这是整个运输层最为经典且必考的核心逻辑。

核心概念: TCP 拥塞控制的三大阶段

  1. 慢启动 (Slow Start)：连接刚建立时，cwnd 通常初始化为 1 个 MSS（最大报文段长度）。在慢启动阶段，每收到一个正确的 ACK，cwnd 就增加 1 个 MSS。这导致发送速率在每个 RTT 内呈指数级增长（1, 2, 4, 8...）。
  2. 拥塞避免 (Congestion Avoidance)：当 cwnd 达到一个特定的阈值（ssthresh，慢启动阈值）时，指数增长将显得过于危险。此时 TCP 进入拥塞避免状态，每个 RTT 只将 cwnd 线性增加 1 个 MSS。
  3. 快速恢复 (Fast Recovery)：针对收到 3 个冗余 ACK 触发的机制。在这个状态下，TCP 尝试在等待丢失的报文段被修复的同时，继续维持一定的发送速率，而不是直接将 cwnd 降到谷底。

易错辨析: TCP Tahoe 与 TCP Reno 的致命辨析

面对丢包事件，早期的 Tahoe 版本和现代广泛部署的 Reno 版本采取了极为不同的惩罚策略：

  • TCP Tahoe（盲目一刀切）：不论丢包是因为超时引起的，还是因为3个冗余 ACK引起的，Tahoe 都会毫不留情地将 cwnd 降为 1 个 MSS，更新 ssthresh，并重新进入慢启动。
  • TCP Reno（现代标准，AIMD 思想）：
    • 如果是因为超时引起丢包（说明网络拥塞极度严重，连 ACK 都回不来了），Reno 的做法和 Tahoe 一样：cwnd 降为 1，重新进入慢启动。
    • 如果是因为3个冗余 ACK引起丢包（说明网络虽然发生拥塞导致个别包掉落，但还能传输后续的分组触发冗余 ACK），Reno 会执行 快速恢复。它将 cwnd 减半（即 \texttt{cwnd = cwnd / 2}，同时将 ssthresh 更新为减半后的值），然后进入拥塞避免（线性增长）状态。

Reno 的这种“遇到冗余 ACK 减半，平时无事则线性增加”的行为，被称为 加性增、乘性减 (AIMD, Additive-Increase, Multiplicative-Decrease)。反映在网络吞吐量随时间变化的图表上，呈现出经典的“锯齿状”行为。

核心概念: TCP 的公平性保障

现实中的挑战： 然而，如果在应用层（如并发的 Web 浏览器）恶意开启几十个并行的 TCP 连接，依然可以无耻地抢占不成比例的带宽份额。此外，由于 UDP 协议是不进行拥塞控制的，因此 UDP 的多媒体流量有时会严重挤占具有“绅士风度”退让机制的 TCP 带宽。

运输层功能的演化 ​

虽然 TCP 和 UDP 在很长一段时间内统治了因特网的运输层，但随着现代应用（特别是 Web 应用）对极低延迟和更高安全性的苛刻要求，运输层的功能正在发生演化。

核心概念: QUIC 协议：现代运输层的革命

在 HTTP/3 标准中，引入了一个全新的应用层协议——QUIC (Quick UDP Internet Connections)。 虽然 QUIC 在技术上被实现在应用层，但它本质上提供的是运输层服务。QUIC 放弃了底层的 TCP，而是建立在 UDP 之上。

  • 面向连接与安全性集成：QUIC 将传统的 TCP 握手和 TLS 加密握手合并，极大地减少了建立安全连接所需的 RTT 数量（甚至是 0-RTT 恢复）。
  • 无队头阻塞的多路复用：在传统 TCP 上的 HTTP/2 虽然实现了多路复用，但如果 TCP 丢失一个报文段，整个 TCP 连接都会被阻塞等待重传，导致所有 HTTP 流受牵连（TCP 级别的队头阻塞）。QUIC 为每个应用层流独立维护状态，一个流的丢包绝不会阻塞其他流的交付。
  • 底层的可靠性与拥塞控制：QUIC 自己在 UDP 之上实现了一套极其类似于 TCP Reno/CUBIC 的可靠数据传输和拥塞控制机制。

网络层：数据平面 ​

运输层负责进程到进程的通信，而网络层 (Network Layer) 则负责将数据报 (Datagram) 从一台主机（源）跨越复杂的网络基础设施，移动到另一台主机（目的）。 在第8版教材中，网络层被分为了两章：本章聚焦于数据平面 (Data Plane)（路由器如何处理到达的分组），下一章聚焦于控制平面 (Control Plane)（路由器如何决定分组的路径）。

网络层概述 ​

网络层存在于因特网中的每一台主机和路由器中。

转发和路由选择：数据平面和控制平面 ​

网络层的作用看似简单（把分组从一端搬到另一端），但实际上为了完成这个任务，网络层必须实现两个极其重要且截然不同的功能。

易错辨析: 转发 (Forwarding) vs. 路由选择 (Routing)

这是网络层最核心的辨析，绝对不能混淆：

  1. 转发 (Forwarding)：
  • 性质：局部、极短时间尺度的硬件动作。
  • 功能：当一个分组到达路由器的某条输入链路时，路由器必须将其转移到正确的输出链路。
  • 发生位置：在路由器的数据平面 (Data Plane) 中，通常由硬件（如 ASIC 芯片）以纳秒级的时间执行。
  1. 路由选择 (Routing)：
  • 性质：全局、较长时间尺度的软件动作。
  • 功能：当分组从发送方流向接收方时，网络必须决定这些分组所采用的端到端路径。计算这些路径的算法被称为路由选择算法 (Routing Algorithm)。
  • 发生位置：在路由器的控制平面 (Control Plane) 中，通常由软件以毫秒或秒级的时间执行。

类比：转发就像是你开车到了一个十字路口，决定从哪个路口开出去；路由选择就像是你出发前在家里用 GPS 规划从北京到上海的整体路线图。

核心概念: 控制平面的两种实现方法

  • 传统的每路由器控制 (Per-router control)：每台路由器都包含一个独立的路由选择组件，它们在控制平面中与其他路由器中的组件相互通信（如 OSPF 或 BGP 协议），共同计算出各自的转发表 (Forwarding Table)。
  • 软件定义网络 (SDN, Software-Defined Networking)：控制平面被物理分离出去，集中在一个远程的 SDN 控制器 中计算。路由器本身变成了一个纯粹的“哑”转发设备，只负责接收来自远程控制器的流表 (Flow Table) 并执行转发动作。

网络服务模型 ​

当网络层将分组从发送方边缘传送到接收方边缘时，它能提供什么服务？例如：它能保证分组一定到达吗？能保证到达的顺序吗？能保证端到端的延迟不超过 100ms 吗？能保证提供 10Mbps 的吞吐量吗？

易错辨析: 因特网的网络服务模型：尽力而为

与传统的 ATM 或帧中继网络不同，因特网的网络层（即 IP 协议）提供的是一种名为 尽力而为服务 (Best-effort service) 的单一服务模型。 名字听起来很好听，但实际上它的含义是：因特网的底层什么都不保证！

  • 不保证交付（可能丢包）。
  • 不保证按序交付（可能乱序）。
  • 不保证端到端时延。
  • 不保证最小带宽。

正是因为底层网络层的“不作为”和极简设计，使得路由器能够高速运转。那些需要可靠性、顺序和拥塞控制的应用，必须依靠位于边缘系统的运输层 (TCP) 自己去实现！

路由器工作原理 ​

要理解数据平面，必须深入路由器的黑盒内部。一台通用的路由器主要由四个组件构成：

  1. 输入端口 (Input ports)：执行物理层、链路层功能，并在网络层执行查找与转发操作。
  2. 交换结构 (Switching fabric)：路由器的核心“大脑”，将路由器的输入端口物理连接到输出端口。
  3. 输出端口 (Output ports)：存储从交换结构接收的分组，并执行链路层和物理层功能将分组发送到输出链路上。
  4. 路由选择处理器 (Routing processor)：执行控制平面功能（运行路由协议、维护路由表，或者在 SDN 中负责与远程控制器通信）。

输入端口处理和基于目的地转发 ​

输入端口中最核心的网络层功能是使用转发表查找输出端口。

核心概念: 最长前缀匹配规则 (Longest Prefix Match)

在传统的基于目的地转发的路由器中，转发表并不为互联网上的每一个单一 IP 地址（约40亿个）保留一条记录，而是存储一系列的地址前缀 (Address Prefixes)。 当一个分组到达时，路由器提取其 32 位目的 IP 地址，并与转发表中的前缀进行匹配。 如果该地址与多个前缀都匹配，路由器将遵循 最长前缀匹配规则：即选择与目的地址匹配的最长的那个前缀所对应的输出链路！

例题: 最长前缀匹配的实战查找

假设转发表中有如下三条表项：

  • 前缀 1：\texttt{11001000 00010111 00010} 链路接口 0
  • 前缀 2：\texttt{11001000 00010111 00010110} 链路接口 1
  • 前缀 3：\texttt{11001000 00010111} 链路接口 2
  • 其他 (Default)： 链路接口 3

如果到达一个分组，其目的 IP 地址的前 24 位是 \texttt{11001000 00010111 00010110}。 解析：这个地址与前缀 1、2、3 全部匹配！但前缀 2 匹配了 24 位，是最长的匹配。因此，路由器会毫不犹豫地将该分组转发到链路接口 1。

为了达到线速转发（每秒处理数亿个分组），这种查找必须在**硬件层（通常是 TCAM，三态内容可寻址存储器）**中以纳秒级的速度完成。

交换 ​

交换结构 (Switching Fabric) 是路由器的核心，它决定了路由器将分组从输入端口转移到输出端口的能力。

核心概念: 交换结构的三种经典类型

  1. 经内存交换 (Switching via memory)：最古老、最简单的路由器。输入端口将分组复制到路由选择处理器的内存中，处理器查表后将其复制到输出端口。交换速度受限于内存带宽（每个分组必须经过总线两次）。
  2. 经总线交换 (Switching via a bus)：输入端口通过一根共享总线将分组直接传送到输出端口，不需要处理器的干预。但由于总线是共享的，同一时刻只能有一个分组跨越总线，总线带宽成为了路由器的性能瓶颈。
  3. 经互联网络交换 (Switching via an interconnection network)：为了突破总线带宽的限制，采用复杂的纵横式交换机（Crossbar Switch，类似多车道的立体交叉网络）。它可以并行地将多个分组同时从不同的输入端口传输到不同的输出端口，是当今高性能核心路由器（如 Cisco CRS 系列）的主流选择。

排队 ​

在路由器的输入端口和输出端口都可能发生排队。如果队列增长得太快，导致路由器的缓存耗尽，就会发生丢包 (Packet Loss)。

核心概念: 输入排队与队头阻塞 (HOL Blocking)

如果交换结构的处理速度不能快到足以无延迟地将所有到达的分组通过交换结构传送，输入端口就会出现排队。 在采用纵横式交换机的路由器中，存在一种极为著名的 队头阻塞 (Head-of-the-Line, HOL blocking) 现象：在一个输入队列中排队的分组必须等待通过交换结构，因为它被位于队列最前部的一个分组阻塞了，即使它所请求的目的输出端口当前是空闲的！

核心概念: 输出排队与分组调度

当交换结构将分组推送到某一个输出端口的速度，超过了该输出链路的物理传输速度时，输出端口就会出现排队。 路由器必须决定如何管理这里的队列：

  • 丢弃策略 (Drop Policy)：当缓存已满时，通常采用 弃尾 (Drop-tail) 策略丢弃新到达的分组；现代路由器常采用 主动队列管理 (AQM)，如随机早期检测 (RED)，在队列完全填满前，根据概率主动丢弃部分分组，以向发送方（如 TCP）提前发出拥塞信号。
  • 分组调度 (Packet Scheduling)：决定队列中哪个分组将被下一个发送出链路。经典机制包括：
    • FIFO (先进先出)：最简单的排队规则。
    • 优先权排队 (Priority Queuing)：高优先级的队列绝对优先发送（如将实时的 VoIP 语音包优先于普通 Web 数据发送，以保证低延迟）。
    • 循环和加权公平排队 (WFQ, Weighted Fair Queuing)：不采用绝对优先，而是保证不同类别的流量都能获得公平或按权重分配的最小带宽保障。

网际协议：IPv4、寻址、IPv6 及其他 ​

网络层的协议栈中，最耀眼的明星无疑是 网际协议 (IP)。

IPv4 数据报格式 ​

网络层分组被称为数据报 (Datagram)。IPv4 数据报的首部包含许多决定网络核心如何处理该分组的关键字段，其典型首部大小为 20 字节（与 TCP 首部相同，这意味着一个没有任何应用层数据的 TCP/IP 报文段就有 40 字节的开销）。

核心概念: IPv4 首部的核心字段

  • 版本号（4 bits）：规定了数据报的 IP 协议版本（即 4）。路由器借此确定如何解释剩余的 IP 数据报。
  • 首部长度（4 bits）：由于 IPv4 首部可以包含可变长的选项字段，该字段确定了实际数据（有效载荷）的起始位置。
  • 服务类型 (TOS)（8 bits）：用于使不同类型的 IP 数据报能相互区别（例如，区分要求低延迟的实时流量和普通的 FTP 流量）。
  • 数据报长度（16 bits）：IP 数据报的总长度（首部加上数据），理论最大值为 65535 字节。
  • 标识、标志、片偏移：这三个字段全部用于极其重要的 IP 数据报分片 (Fragmentation) 功能。
  • 寿命 (TTL, Time-to-Live)（8 bits）：用来确保数据报不会在网络中的路由环路里永远循环。每当数据报经过一台路由器时，TTL 的值减 1。若 TTL 减为 0，路由器必须无情地丢弃该数据报。
  • 协议（8 bits）：指示了该 IP 数据报的数据部分应该交给目的主机上的哪个特定的运输层协议（如 6 代表 TCP，17 代表 UDP）。该字段是将网络层和运输层绑定的纽带。
  • 首部检验和（16 bits）：帮助路由器检测收到的 IP 数据报中的首部（仅限首部，不包括数据部分）是否发生比特差错。
  • 源和目的 IP 地址（各 32 bits）：网络层寻址的核心。

易错辨析: IP 分片与重组的陷阱

并不是所有的链路层协议都能承载相同长度的网络层分组（例如，以太网帧通常最多承载 1500 字节的有效载荷，这被称为最大传输单元 MTU）。如果一个庞大的 IP 数据报必须经过一条 MTU 较小的链路，路由器就必须将其分片 (Fragment)，分成几个较小的数据报。 核心辨析点：分片是在网络核心（路由器）中进行的，但为了保持网络核心的简单高效，重组 (Reassembly) 工作绝对不会在任何路由器中进行，而是**全部交由目的端系统（接收方主机）**完成！ 注：由于分片带来了巨大的性能开销和严重的安全隐患，如今的协议设计都在极力避免网络内部分片（例如 IPv6 已经彻底禁止了路由器的分片行为）。

IPv4 编址 ​

主机与物理链路之间的边界被称为接口 (Interface)。一台主机通常只有一个或两个接口（如网线接口和 WiFi 接口），而一台路由器通常拥有多个接口。必须牢记的是，IP 地址实际上是分配给接口的，而不是直接分配给主机的。

核心概念: 子网 (Subnet)

IP 编址规则规定，连接在同一条物理网络上（且它们之间不经过任何路由器）的所有设备接口，必须拥有相同的 IP 地址网络部分。这个由共享相同网络前缀的接口组成的隔离网络孤岛，就被称为子网。 为了在复杂的拓扑图中确定子网，你可以将所有路由器的接口从与其相连的链路上“切断”，形成一个个孤立的网络。每一个这样的隔离网络就是一个子网。

DHCP：动态主机配置协议 ​

如果一个组织获得了一块地址（例如一个 的子网），它如何将具体的 IP 地址分配给内部成百上千的主机呢？静态手动配置极为繁琐。今天，几乎所有的接入网络都使用了 动态主机配置协议 (DHCP) 来实现即插即用。

例题: DHCP 的“发现-提供-请求-确认” (DORA) 过程

DHCP 是一个客户-服务器协议。当一台新电脑（客户端）接入网络时，为了获取 IP 地址，它会经历经典的四个步骤：

  1. DHCP 发现 (Discover)：新电脑在不知道任何网络信息的情况下，向子网的受限广播地址 \texttt{255.255.255.255} 盲发一个 UDP 发现报文，其源 IP 只能设为 \texttt{0.0.0.0}。
  2. DHCP 提供 (Offer)：收到发现报文的 DHCP 服务器（网络中可能存在多个），向客户端发回一个广播的提供报文，其中包含一个推荐的 IP 地址、网络掩码以及 IP 地址租用期 (Lease time)。
  3. DHCP 请求 (Request)：客户端从一个或多个提供中选择一个，向选定的服务器发送一个 DHCP 请求报文，回显所请求的配置参数。
  4. DHCP ACK (确认)：服务器用 ACK 报文对请求进行响应，证实所要求的参数。此时，客户端正式获取了 IP 地址。

极其重要：通过 DHCP，主机不仅得到了 IP 地址，还获取了另外两个绝对不可或缺的信息：第一跳路由器（默认网关）的 IP 地址，以及 本地 DNS 服务器的 IP 地址！

网络地址转换 (NAT) ​

在上一小节中，我们探讨了 DHCP 如何为子网内的主机动态分配 IP 地址。然而，面对全球迅速耗尽的 IPv4 地址空间，单靠 DHCP 是远远不够的。为了让家庭或小型企业网络中的多台设备共享一个全球唯一的公共 IP 地址，网络地址转换 (NAT, Network Address Translation) 技术应运而生。

易错辨析: NAT 的巨大争议与对架构原则的破坏

尽管 NAT 拯救了 IPv4 并被极其广泛地部署，但它遭到了许多网络体系结构纯粹主义者的强烈反对，这是各大考试中的高频辨析题：

  • 违反了层次原则：路由器作为网络层（第 3 层）设备，理应只处理网络层的 IP 地址。但 NAT 却极其粗暴地修改了运输层（第 4 层）的端口号！
  • 违背了端到端原则 (End-to-End Argument)：主机应当直接相互通信，而不应该由中间节点修改 IP 地址和端口号。
  • 严重阻碍了 P2P 应用程序：在 P2P 网络（如 BitTorrent 或 Skype）中，对等方需要主动发起连接。但由于处于 NAT 后面的主机没有公共 IP 地址，且无法接收未经请求的入站连接，这就迫使开发者不得不设计极其复杂的 NAT 穿越 (NAT Traversal) 或打洞技术。

结论：NAT 只是一个应对 IPv4 地址枯竭的“权宜之计”，解决该问题的终极方案是全面迁移到具有无尽地址空间的 IPv6。

IPv6 ​

为了彻底解决 32 位 IPv4 地址空间即将耗尽的问题（并在设计上吸取 IPv4 的教训），IETF 研发了 IPv6。

核心概念: IPv6 数据报格式的革命性改进

IPv6 将 IP 地址长度从 32 位增加到了 128 位，这确保了地球上每一粒沙子都能分到一个 IP 地址。此外，它对首部进行了极大的精简，引入了固定的 40 字节基本首部。

与 IPv4 相比，IPv6 的核心变化如下：

  • 扩大的地址容量：引入了单播 (unicast)、多播 (multicast) 以及全新的任播 (anycast) 地址（将数据报交付给一组主机中距离最近的一个）。
  • 流标签 (Flow Label)：新增了 20 比特的流标签字段，用于给特定应用（如高质量视频流）的数据报打上标签，以便路由器提供特殊的服务质量 (QoS)。
  • 取消了分片/重新组装 (Fragmentation / Reassembly)：IPv6 彻底禁止路由器对数据报进行分片！如果一个 IPv6 数据报太大而无法通过某条链路，路由器会直接丢弃它，并向发送方回送一个“分组太大”的 ICMPv6 差错报文。发送方收到后会使用较小的 MSS 重新发送。这极大地加快了路由器的处理速度。
  • 取消了首部检验和 (Header Checksum)：由于运输层（TCP/UDP）和链路层（以太网）都已经执行了检验和操作，网络层的检验和被认为是多余的，取消它能省去路由器每跳都需要重新计算检验和的开销。
  • 选项 (Options) 字段的移除：选项字段不再属于标准的首部，而是出现在 IPv6 首部指出的 下一个首部 (Next Header) 指向的位置。

例题: 从 IPv4 向 IPv6 的迁移策略：建隧道 (Tunneling)

目前因特网中充斥着海量的 IPv4 路由器，它们无法直接识别 IPv6 数据报。我们不可能在一夜之间将所有路由器升级，因此广泛采用了一种极其重要的过渡机制：建隧道。 假设两台 IPv6 路由器（B 和 E）之间隔着几台仅支持 IPv4 的路由器（C 和 D）。

  1. 路由器 B 拿到一个完整的 IPv6 数据报，它知道接下来的路径是纯 IPv4 网络。
  2. B 会将整个 IPv6 数据报作为有效载荷，封装到一个新的 IPv4 数据报的实体体中，目的地址设为 E 的 IPv4 地址。
  3. 路由器 C 和 D 就像传递普通 IPv4 分组一样将其一路转发，完全不知道里面其实藏着一个 IPv6 数据报。
  4. 路由器 E 收到该 IPv4 数据报后，剥离 IPv4 首部，取出内部的 IPv6 数据报并继续按照 IPv6 规则路由。

泛化转发和 SDN ​

在前面的小节中，我们学习的是传统的基于目的地转发 (Destination-based Forwarding)，即路由器仅仅根据数据报的目的 IP 地址，在转发表中进行最长前缀匹配。 然而，现代网络（特别是软件定义网络 SDN）引入了一种极其强大的新范式：泛化转发 (Generalized Forwarding)。

匹配加操作 (Match-plus-action) ​

在泛化转发中，转发决策不再仅仅局限于网络层的目的 IP 地址。交换机可以基于网络栈中第二层、第三层、甚至第四层的首部字段的任意组合来进行匹配并采取行动。最著名的泛化转发标准是 OpenFlow 协议。

核心概念: OpenFlow 流表 (Flow Table) 与匹配原则

在 SDN 架构中，远程控制器会计算出流表，并将其下发给底层的“哑”交换机。流表的每个表项包含：

  • 首部字段值的集合 (Match Values)：入站分组将与这些值进行匹配。
  • 计数器集合 (Counters)：如匹配该表项的分组数量和上次更新以来的时间。
  • 操作集合 (Actions)：当分组匹配时要执行的操作。

交换机在收到分组时，可以执行极为精细的匹配，例如同时匹配：源 MAC 地址、目的 IP 地址、协议类型（如 TCP）、以及 目的端口号（如 80）。

操作 (Actions) ​

传统的路由器匹配后只能“转发”，而支持 OpenFlow 泛化转发的交换机在匹配成功后可以执行丰富的操作 (Action)：

易错辨析: 泛化转发的四大核心操作

  1. 转发 (Forward)：将分组转发到一个特定的输出端口，或通过多播转发到多个端口。
  2. 丢弃 (Drop)：如果不满足任何规则，或者为了实现防火墙过滤功能（如丢弃所有目的端口为 22 的分组），可以直接丢弃。
  3. 修改字段 (Modify-field)：在将分组转发出去之前，重写其首部字段（例如，重写源/目的 IP 地址和端口号，这就完美实现了 NAT 的功能！）。
  4. 向控制器发送 (Send-to-controller)：如果分组未能与流表中的任何条目匹配，交换机可以将其打包发送给远程控制器，由控制器决定如何处理，并随后下发新的流表规则。

通过这种“匹配加操作”的抽象机制，一台物理上的 OpenFlow 交换机，可以瞬间被配置为一台路由器（根据目的 IP 转发）、一台交换机（根据目的 MAC 转发）、一台防火墙（根据 IP/端口拦截），甚至是一台NAT 设备（重写 IP 和端口）。这种将功能彻底解耦和可编程化的能力，正是 SDN 颠覆传统网络架构的核心所在。

中间盒 ​

随着因特网的发展，网络层不仅需要进行传统的寻址和转发，还引入了许多执行其他功能的设备，这些设备统称为中间盒 (Middlebox)。

核心概念: 中间盒的定义与典型代表

中间盒是指在源主机和目的主机之间的数据路径上，执行除了标准IP路由和转发之外的其他功能的任何中间实体。 常见的中间盒包括：

  • NAT 转换器：重写数据报的源/目的IP地址和端口号，解决IPv4地址短缺问题并隐藏内部网络结构。
  • 防火墙 (Firewall) 与入侵检测系统 (IDS)：根据特定的规则（如五元组、报文行为特征）拦截、丢弃或记录恶意的分组，保障网络安全。
  • 负载均衡器 (Load Balancer)：将到达某个公共IP的大量服务请求，根据当前网络负载情况，动态地分发到数据中心内多台不同的后端服务器上。

易错辨析: 中间盒带来的争议与挑战

中间盒在当今因特网中无处不在，但它们极大地破坏了因特网最初的端到端原则 (End-to-End Principle)。传统上，网络核心应当保持极度简单（只负责尽力而为的IP转发），所有的复杂性（如连接状态、应用层逻辑）应当留在网络边缘。但中间盒的引入使得网络核心变得“有状态”且极其复杂，这导致许多新型应用（如P2P文件分发和某些实时通信）的部署面临巨大阻碍（例如不得不设计极其复杂的NAT穿越技术）。

网络层：控制平面 ​

在第4章中，我们学习了数据平面（即路由器如何将分组从输入接口快速转移到输出接口）。在本章中，我们将聚焦于控制平面 (Control Plane)，即网络是如何计算和维护这些决定分组去向的转发表（或流表）的。

概述 ​

控制平面主要解决一个核心问题：如何计算出从源端到目的端的良好路径？ 计算这些路径（转发表）通常有两种根本不同的方法：

  1. 每路由器控制 (Per-router control)：这是传统的路由选择算法。每台路由器中都包含一个相对独立的路由选择组件，各个路由器的该组件通过相互通信（交换路由更新报文）来分布式地计算出各自的转发表。OSPF和BGP协议就是基于这种方法。
  2. 逻辑集中的控制 (Logically centralized control)：这是软件定义网络(SDN)的核心理念。一个物理上分离的、逻辑上集中的远程控制器负责掌握全局状态并计算转发表（或流表），然后通过特定协议（如OpenFlow）将其下发给底层的各个“哑”路由器执行。

路由选择算法 ​

无论是传统的每路由器控制，还是SDN集中的控制器，其计算路径的核心本质都是路由选择算法 (Routing Algorithm)。其目的是给定一组路由器以及连接它们的链路，找出一条从源路由器到目的路由器的“好”路径（通常指具有最低成本的路径）。

为了进行数学计算，我们将网络抽象为一个图 (Graph) ，其中是节点（路由器）的集合，是边（链路）的集合。每条边都有一个相关联的成本。

路由选择算法可以通过多种方式进行分类，其中最核心的分类方式是根据算法是全局的还是分布式的。

链路状态路由选择算法 ​

链路状态 (Link-State, LS) 算法是一种全局路由选择算法。在这种算法中，网络拓扑和所有的链路成本都是已知的，即算法在开始计算最短路径之前，必须先获得整个网络的完整信息。

核心概念: LS算法的核心工作机制

在网络实践中，这通常通过让每个节点向网络中的所有其他节点广播链路状态分组来实现（即链路状态广播 (Link-State Broadcast)）。当所有广播完成后，每个节点最终都会构建出一个完全相同的、全网一致的网络拓扑图。 随后，每个节点在自己的本地内存中独立运行Dijkstra（迪杰斯特拉）算法，计算出从它自己到所有其他目的节点的最低成本路径，从而生成转发表。

易错辨析: DV算法的“无穷计数”问题 (Count-to-Infinity)

这是DV算法中最臭名昭著的缺陷，也是各大考试的高频考点： 当链路成本减少（即好消息）时，好消息会在网络中传播得非常快。 但是，当某条链路成本发生暴增或链路彻底断开（即坏消息）时，由于节点只知道到达目的地的“数值距离”，而根本不知道具体的“完整路径”，节点之间可能会互相误导，产生致命的路由环路 (Routing Loop)。 例如：节点A原本通过节点B到达目的节点C。当B到C的链路突然断开时，B检查A发来的距离向量，发现A宣称它能到达C（B并不知道A其实是打算经过B自己的）。B误以为A找到了一条新路，于是更新自己的路由表指向A。A接着发现B的成本增加了，又更新自己指向B。两者的距离向量成本不断互相累加，直到达到无穷大（Infinity），在此期间所有发往C的分组都将在A和B之间无限循环！

核心概念: 毒性逆转 (Poisoned Reverse)

为了解决特定情况下的路由环路问题，DV算法引入了毒性逆转机制作为补救措施。 其核心思想是：如果节点A通过节点B路由到达目的地C，那么A将向B隐瞒真实成本，反而向B宣称它到达C的距离为无穷大 ()。 由于B认为A到达C的成本是无穷大，所以只要A依然通过B路由到C，B就绝对不会试图将数据发给A来路由到C，从而打破了上述的两节点环路。 局限性：毒性逆转只能解决涉及两个直接相连节点的简单路由环路，对于涉及三个或更多节点的复杂环路依然无能为力。

例题: LS算法与DV算法的终极对比

在实际构建网络时，该如何在这两种经典算法中做出选择？

  • 报文复杂性：LS算法需要向全网广播链路状态信息，每次某条链路状态发生变化都会引发全网的风暴式广播，通信开销极大；DV算法只在直接相连的邻居之间交换信息，报文交互量相对较小。
  • 收敛速度：LS算法的收敛速度较快（仅需的计算时间），且不会产生选路环路；DV算法收敛较慢，且极其容易遇到路由环路和无穷计数问题。
  • 健壮性：如果一台路由器发生硬件故障或被黑客攻破。在LS算法中，该路由器只能广播关于它自己相连链路的错误成本，全网计算是各自独立的，因此提供了一定的隔离性；而在DV算法中，一个节点计算的错误距离向量会直接传递给邻居，并继续在整个网络中级联扩散，因此一个错误配置的节点就有可能搞垮整个网络的路由！

因特网中自治系统内部的路由选择：OSPF ​

前一节讨论的路由选择算法（LS 和 DV）都假设网络是一个由同质路由器组成的单一的、扁平的网络，这显然不符合实际情况。因特网是由数以万计的异构网络组成的“网络的网络”，不可能让全球所有路由器运行同一个路由选择算法。

核心概念: 自治系统 (Autonomous System, AS)

为了解决规模扩展和管理自治的问题，因特网将路由器组织进自治系统 (AS)。

  • 定义：一个 AS 是由同一个管理机构（如 ISP、大型企业或大学）控制下的一组路由器。每个 AS 都由一个唯一的 AS 号 (ASN) 标识。
  • 内部网关协议 (IGP, Interior Gateway Protocol)：在同一个 AS 内部运行的路由选择算法。同一 AS 内的所有路由器必须运行相同的 IGP。
  • 外部网关协议 (EGP, Exterior Gateway Protocol)：负责在不同的 AS 之间路由分组的协议（如 BGP）。

OSPF 协议简介 ​

OSPF (开放最短路径优先, Open Shortest Path First) 是因特网中最广泛使用的 IGP 之一。顾名思义，“开放”意味着其规范是公开的（RFC 2328），而“最短路径优先”则表明它是一种基于 Dijkstra 算法的链路状态 (LS) 协议。

核心概念: OSPF 的核心机制

  • 链路状态广播：OSPF 路由器向 AS 内的所有其他路由器广播链路状态信息（不仅是向相邻路由器）。每次链路状态发生变化（或周期性地），路由器都会洪泛 OSPF 链路状态通告 (LSA)。
  • 拓扑数据库：每个 OSPF 路由器都维护一个完整的、关于整个 AS 拓扑的数据库（即所有链路状态的集合）。
  • 独立计算：基于这个本地拓扑数据库，每个路由器独立运行 Dijkstra 算法，计算出以自己为根的、到达所有子网的最短路径树。
  • 直接封装在 IP 中：OSPF 报文不使用 TCP 或 UDP，而是直接封装在 IP 数据报中（协议号为 89）。OSPF 自身实现了诸如可靠报文传输和链路状态广播的机制。

易错辨析: OSPF 的高级特性

作为一种成熟的工业级路由协议，OSPF 包含了许多我们在简单 LS 算法中未提及的高级特性：

  1. 安全性：所有 OSPF 报文都可以被鉴别 (Authenticated)，防止恶意路由器注入虚假的链路状态信息。
  2. 多条相同成本的路径：当存在多条成本相同的路径到达同一目的地时，OSPF 允许负载均衡，而不是仅仅选择其中一条。
  3. 对单播和多播路由的综合支持：多播 OSPF (MOSPF) 扩展了 OSPF 以支持多播路由。
  4. 支持在单个 AS 内的层次结构：为了应对极大规膜的 AS，OSPF 允许将 AS 划分为多个区域 (Area)（包括一个骨干区域和若干个局部区域）。链路状态广播仅在区域内部进行，这极大地减少了全网洪泛的开销。

ISP 之间的路由选择：BGP ​

当分组的目的地在当前的 AS 之外时，我们需要一种跨越 AS 边界的路由协议。因特网中所有的 AS 都使用同一种外部网关协议：边界网关协议 (BGP, Border Gateway Protocol)。可以说，BGP 是将因特网粘合在一起的“超级胶水”。

BGP 的作用与原理 ​

核心概念: BGP 的双重角色

对于一个位于 AS 边缘的路由器（即网关路由器 Gateway Router），BGP 为其提供两种功能：

  1. eBGP (外部 BGP)：从相邻的 AS 处获得子网可达性信息。
  2. iBGP (内部 BGP)：将获得的子网可达性信息传播给该 AS 内部的所有路由器。

通过这两种机制，每个路由器都能知道（1）哪些外部子网是可达的，以及（2）到达这些外部子网应该通过哪个网关路由器离开本 AS。

易错辨析: BGP 是路径向量 (Path Vector) 协议

虽然 BGP 和 DV 算法一样都是分布式、异步交换信息的，但 BGP 并不交换简单的“成本数值”。 在 BGP 中，每个 AS 在通告一个子网前缀时，不仅会说“我可以到达子网 X”，还会附带上它到达该子网所经过的完整的 AS 序列 (AS-PATH)。 例如，AS1 通告它能到达子网 X，AS-PATH 可能为 \texttt{AS1}；AS2 收到后，向 AS3 通告时，AS-PATH 会变成 \texttt{AS2, AS1}。 这种路径向量机制是 BGP 解决“无穷计数”和路由环路的关键：如果一台 BGP 路由器在接收到的 AS-PATH 中看到了自己的 AS 号，它会立即拒绝该通告，从而从根本上杜绝了环路！

BGP 路由选择与策略 ​

当一台 BGP 路由器得知了多条到达同一个目的子网的不同路径时，它必须选择其中一条。BGP 的路由选择过程是一个复杂的决策序列。

关键公式: BGP 路由选择决策过程 (按优先级执行)

  1. 本地偏好 (Local Preference)：这是一个由网络管理员手动配置的策略属性。具有最高本地偏好值的路径被优先选择。
  2. 最短 AS-PATH：如果本地偏好相同，选择经过 AS 数量最少（即 AS-PATH 最短）的路径。
  3. 热土豆路由 (Hot Potato Routing)：如果前两个条件仍相同，路由器会选择到达下一个 AS 的网关路由器成本最低（由内部 IGP 如 OSPF 决定）的路径。其核心理念是：“尽快把分组踢出我的 AS，以节省我自己的内部资源”。
  4. BGP 标识符：如果上述条件全部相同，通常使用 BGP 标识符（如路由器 IP）来打破平局。

例题: BGP 策略 (Policy) 胜过一切性能指标

在 AS 内部（OSPF），路由选择的首要目标是性能（如寻找最快的路径）。 然而，在 AS 之间（BGP），路由选择的绝对主导因素是策略 (Policy)，尤其是经济利益！ 例如：ISP A 是 ISP B 的客户。ISP B 也是 ISP C 的客户。 当 B 收到一个发往外部网络的分组时，如果它把分组路由给 C，B 必须向 C 付费。如果 B 把它路由给另一个与它对等 (Peering) 的 ISP，可能不需要付费。 因此，即使通过 C 的物理路径更短、速度更快，B 的网络管理员也极有可能配置 BGP 策略（修改本地偏好），强行迫使流量走免费的对等链路，而不是走昂贵的客户链路。在 BGP 的世界里，钱比速度更重要。

SDN 控制平面 ​

在传统的路由选择方法中，每台路由器既包含数据平面（转发硬件），又包含控制平面（运行 OSPF、BGP 等路由选择协议的软件）。而 软件定义网络 (SDN, Software-Defined Networking) 彻底打破了这种紧耦合，将控制平面从物理路由器中剥离出来，集中到了远程的控制器中。

核心概念: SDN 体系结构的四个关键特征

  1. 基于流的转发 (Flow-based forwarding)：SDN 交换机的转发决策基于传输层、网络层或链路层首部中任意数量字段的组合（即流表中的匹配加操作规则），而不仅仅是目的 IP 地址。
  2. 数据平面与控制平面分离：数据平面由一组互联的“哑”交换机组成（只负责高速转发），而控制平面则由服务器上的软件构成，负责计算和下发流表。
  3. 网络控制功能位于数据平面交换机外部：控制平面分为两个部分：SDN 控制器 (SDN Controller)（维护网络状态的通用平台）和 网络控制应用程序 (Network-control applications)（实现具体路由、负载均衡或防火墙逻辑的上层业务软件）。
  4. 可编程的网络 (Programmable network)：通过在 SDN 控制器提供的 API 之上编写网络控制应用程序，网络管理员可以极其灵活地对网络行为进行编程和定义。

SDN 控制平面：SDN 控制器和 SDN 网络控制应用程序 ​

SDN 控制器是整个网络的大脑，它在宏观上维护着对整个网络状态的精准视图。

易错辨析: SDN 控制器的内部层次结构

SDN 控制器通常被划分为三个逻辑层次：

  • 与网络控制应用程序交互的接口（北向 API）：允许上层应用读取网络状态并表达控制意图（例如提供 RESTful API）。
  • 网络范围状态管理层：控制器的核心。维护诸如网络拓扑、所有链路状态、已连接主机的信息以及各交换机的流表副本等分布式数据库。
  • 通信层（南向协议）：控制器与底层的受控交换机之间进行通信的协议接口（最著名的是 OpenFlow 协议），用于下发流表和接收交换机的状态更新。

OpenFlow 协议 ​

OpenFlow 协议是 SDN 控制器与底层数据平面交换机之间通信的“通用语言”（即南向协议）。它运行在 TCP 之上（通常使用端口 6653）。

核心概念: OpenFlow 报文的三大类

  1. 控制器到交换机的报文：由控制器主动发送，用于查询交换机功能、配置交换机参数或下发/修改/删除流表项（Modify-State 报文）。
  2. 异步报文：由交换机主动发送给控制器。最典型的是 Packet-in 报文，当交换机收到一个无法与现有流表匹配的分组时，会将其封装在 Packet-in 报文中发给控制器请示如何处理。
  3. 对称报文：由任何一方发送，如 Echo（用于保活/心跳检测）和 Hello 报文。

ICMP：因特网控制报文协议 ​

在复杂的网络环境中，路由器或主机在处理 IP 数据报时经常会遇到各种异常（如找不到目的网络、TTL 耗尽等）。因特网控制报文协议 (ICMP, Internet Control Message Protocol) 被主机和路由器用来沟通网络层的信息，其中最典型的用途就是差错报告。

核心概念: ICMP 报文的封装与结构

ICMP 报文不是直接交由链路层传输的，而是作为 IP 数据报的有效载荷进行封装的（就像 TCP 或 UDP 报文段一样，ICMP 的协议号为 1）。 一个 ICMP 报文包含：

  • 类型 (Type) 和 代码 (Code) 字段：共同标识具体的 ICMP 报文含义。
  • 检验和 (Checksum)：用于差错检测。
  • 导致差错的 IP 数据报首部及前 8 个字节：这是极其巧妙的设计！发送方收到 ICMP 差错报文后，可以通过这部分内容精准地判断出到底是自己发送的哪一个上层协议进程（通过提取源/目的端口号）引发了错误。

关键公式: 常见的 ICMP 报文类型与代码

  • Type 0, Code 0：回显回答 (Echo Reply)（Ping 命令的响应）。
  • Type 8, Code 0：回显请求 (Echo Request)（Ping 命令的请求）。
  • Type 3, Code 1：目的主机不可达。
  • Type 3, Code 3：目的端口不可达（通常在目的主机没运行相关应用层服务时返回）。
  • Type 11, Code 0：TTL 过期（当 IP 数据报的寿命字段减为 0 时返回，这是 Traceroute 工作的基础！）。

例题: Traceroute 的奇妙实现原理

我们在第 1 章提到了 Traceroute 程序可以探测到达目的主机所经过的所有路由器。它其实是巧妙地利用了 IP 的 TTL 字段和 ICMP 的报错机制：

  1. 源主机向目的主机发送一系列极其普通的 UDP 数据报（使用一个不可能被使用的极大端口号）。
  2. 第一批数据报的 TTL 被故意设为 1。当它到达第一跳路由器时，TTL 减为 0，第一跳路由器丢弃它，并向源主机发回一个 ICMP TTL过期 (Type 11, Code 0) 报文。源主机借此记录下了第一跳路由器的 IP 地址和 RTT。
  3. 第二批数据报的 TTL 被设为 2。它穿过第一跳，在第二跳路由器处 TTL 耗尽，触发第二跳路由器发回 ICMP TTL过期报文。
  4. 如此递增 TTL，直到某个 UDP 数据报真正到达目的主机。目的主机发现 TTL 未耗尽，但由于目的端口不可达，它会向源主机发回一个 ICMP 端口不可达 (Type 3, Code 3) 报文。
  5. 源主机一旦收到 Type 3 的 ICMP 报文，就知道探测已经到达终点，程序结束。

网络管理、SNMP 和 NETCONF/YANG ​

大型网络（如 ISP 或跨国企业网络）包含数以千计的互联设备（路由器、交换机、服务器）。网络管理员必须有工具来监控这些设备的健康状态、配置参数以及定位故障。这被称为网络管理 (Network Management)。

网络管理框架 ​

核心概念: 网络管理的核心组件

  • 管理服务器 (Managing server)：通常是一个具有用户界面的强大应用程序，由网络管理员操作，是网络管理的“控制中心”。
  • 被管设备 (Managed device)：存在于被管网络中的设备（如路由器、主机）。每个被管设备内部都运行着一个网络管理代理 (Network management agent) 软件进程，负责收集本地状态并与管理服务器通信。
  • 管理信息库 (MIB, Management Information Base)：被管设备中收集和维护的信息（如特定接口丢弃的分组数量、当前 CPU 负载等）被组织成 MIB 对象。
  • 网络管理协议 (Network management protocol)：运行在管理服务器和被管设备之间，用于查询 MIB 数据或修改设备配置。

简单网络管理协议和管理信息库 ​

简单网络管理协议 (SNMP, Simple Network Management Protocol) 是一种广泛使用的、传统的网络管理协议（通常运行在 UDP 端口 161 上）。

易错辨析: SNMP 的两种工作模式：请求-响应 vs. 陷入 (Trap)

  • 请求-响应模式：管理服务器主动向被管设备的代理发送 SNMP Get（获取数据）或 Set（修改配置）请求，代理执行后返回响应。这种方式通常用于例行轮询（Polling）。
  • 陷入 (Trap) 模式：如果被管设备发生紧急异常（如链路突然断开、温度过高），代理不需要等待轮询，而是主动向管理服务器发送一个 SNMP 陷入报文 (Trap message) 进行告警。这极大降低了管理服务器不断轮询产生的巨大网络开销。

NETCONF 和 YANG ​

随着软件定义网络 (SDN) 和网络自动化需求的激增，SNMP 在配置复杂设备时显得力不从心（SNMP 最初设计更偏向于“监控读取”而非“复杂配置写入”）。为此，业界开发了新一代的网络配置管理协议：NETCONF 及其伴生的建模语言 YANG。

核心概念: NETCONF 和 YANG 的现代网络管理

  • NETCONF 协议：提供了一种机制，用于安装、操作和删除网络设备的配置。它克服了 SNMP 的缺点，采用基于 XML 的编码格式来传输复杂的配置数据，并且必须运行在提供机密性和身份验证的**安全的、可靠的运输协议（如 SSH 或 TLS 之上的 TCP）**上，而不再是不可靠的 UDP。
  • YANG 数据建模语言：与协议解耦。YANG 被专门用来定义和描述 NETCONF 客户端和服务器之间交换的配置数据和状态数据的结构（类似于 XML Schema）。它使得全网异构设备的配置数据拥有了统一的、机器可读的严格语法规范，极大推动了网络自动化的进程。

链路层和局域网 ​

在前面的章节中，我们已经沿着协议栈自顶向下，分别研究了应用层、运输层和网络层。网络层（IP协议）负责将数据报从源主机通过一系列路由器路由到目的主机。然而，在端到端的路径上，数据报必须跨越多个独立且物理性质各异的单个链路。本章我们将聚焦于链路层 (Link Layer)，探究它是如何将分组从一个节点无误地移动到相邻节点的。

链路层概述 ​

为了清晰地讨论链路层，我们首先需要统一定义几个核心术语：

  • 节点 (Node)：运行链路层协议的任何设备，包括主机、路由器、交换机和 WiFi 接入点等。
  • 链路 (Link)：沿着通信路径连接相邻节点的通信信道。为了将一个数据报从源移动到目的地，它必须通过沿途的一系列链路。
  • 帧 (Frame)：链路层的协议数据单元。网络层的数据报在通过链路传输之前，必须被封装在链路层帧中。

易错辨析: 旅行类比：链路层与网络层的分工

让我们再次回到旅行的类比。假设你（数据报）计划从北京（源主机）去纽约（目的主机）。 你的行程（路由选择，网络层）是：从北京坐高铁到上海，然后从上海坐飞机到纽约。 在这个过程中，管理高铁线路的公司和管理跨洋航线的航空公司就是链路层协议。 它们提供的是完全不同的物理传输服务（钢轨 vs. 航空），但它们的共同目标是将你安全地从当前节点（北京/上海）移动到下一个相邻节点（上海/纽约）。至于你最终要去哪里，高铁公司和航空公司并不关心，那是你自己的旅行计划（IP 数据报首部）决定的。

链路层提供的服务 ​

尽管任何链路层的基本服务都是将数据报通过单一通信链路从一个节点移动到相邻节点，但具体的链路层协议（如以太网、WiFi）所提供的详细服务可能千差万别。

核心概念: 链路层可能提供的核心服务

  1. 成帧 (Framing)：在每个网络层数据报经链路传送之前，将其用链路层首部和尾部封装起来，构成一个帧。首部通常包含源和目的物理地址 (MAC 地址)。
  2. 链路接入 (Link Access)：由媒体访问控制 (MAC, Medium Access Control) 协议规定帧在链路上何种时刻可以被传输。对于共享的广播链路（如早期的集线器网络或 WiFi），MAC 协议负责协调多个节点的传输以避免碰撞。
  3. 可靠交付 (Reliable Delivery)：保证无差错地经链路移动每个网络层数据报。
  4. 差错检测和纠正 (Error Detection and Correction)：由于信号衰减或电磁噪声，接收方节点可能会收到包含比特翻转（0变1，1变0）的帧。链路层硬件提供极其快速的检测甚至纠正机制。

易错辨析: 可靠交付在链路层的作用与取舍

你可能会问：既然运输层（TCP）已经提供了可靠数据传输，为什么还要在链路层提供？

  • 对于高差错率的链路（如无线 WiFi 链路），链路层提供可靠交付是极其有价值的。它能在本地（发生错误的链路两端）以极快的速度纠正错误，避免触发高昂的端到端 TCP 超时重传。
  • 对于低差错率的有线链路（如光纤、双绞铜线以太网），链路层的可靠交付开销被认为是不必要的。因此，传统的有线以太网不提供可靠交付服务。

链路层在何处实现 ​

链路层绝大部分是在硬件中实现的。 在绝大多数主机中，链路层主体部分是在一块称为 网络适配器 (Network Adapter)，或者更常被称为 网络接口卡 (NIC, Network Interface Card) 的独立芯片上实现的。NIC 的核心是链路层控制器，这是一个专用的 ASIC（专用集成电路） 芯片，它以极快的硬件级速度执行成帧、链路接入、差错检测等核心逻辑。

差错检测和纠正技术 ​

为了保护数据免受物理链路噪声的干扰，发送节点会在帧中附加差错检测和纠正比特（Error-Detection and -Correction bits, EDC）。接收节点通过比对接收到的数据 和 来判断是否发生了错误。

奇偶校验 ​

这是最简单、最古老的差错检测方法。

核心概念: 单比特奇偶校验 (Single-bit Parity)

假设发送方要发送 个数据比特。在偶校验 (Even parity) 方案中，发送方只需附加一个额外的比特（第 个比特），使得这 个比特中“1”的总数是偶数。

  • 优点：计算极其简单。
  • 致命缺点：它只能检测出奇数个比特差错。如果发生了两个比特同时翻转的突发错误，单比特奇偶校验将彻底失效（它会误以为数据完全正确）。

核心概念: 二维奇偶校验 (Two-dimensional Parity)

为了改进单比特奇偶校验，将数据划分为行和列。对每一行和每一列分别计算奇偶校验值。

这种二维结构极其强大：如果报文中出现了一个单比特差错，它将导致包含该比特的行校验和列校验同时失效。 接收方不仅能够检测出差错，还能利用行列的交叉点精准定位并纠正 (Correct) 这个差错！接收方无需请求发送方重传即可自行修复数据，这种能力被称为 前向纠错 (FEC, Forward Error Correction)。

检验和方法 ​

我们在运输层（TCP 和 UDP）中已经深入探讨过 因特网检验和 (Internet Checksum)。 它将数据视为 16 比特的整数序列，求和后取反码。这种方法计算极其轻量，可以用软件高效实现，因此广泛应用于运输层和网络层（IP首部）。 然而，对于链路层而言，检验和方法太弱了。由于物理链路经常发生突发性的、成片的比特翻转，检验和的碰撞概率较高。因此，链路层需要依靠纯硬件实现的、数学上更严密的 CRC 算法。

循环冗余检测 (CRC) ​

当今计算机网络（包括以太网、WiFi）链路层广泛使用的差错检测技术是 循环冗余检测 (CRC, Cyclic Redundancy Check) 编码，也称为 多项式编码 (Polynomial codes)。

多路访问链路和协议 ​

在链路层中，存在两种类型的网络链路：点对点链路 (Point-to-point link) 和 广播链路 (Broadcast link)。

  • 点对点链路：连接一对发送方和接收方（如 PPP 协议）。
  • 广播链路：多个节点连接到同一个共享的通信信道（如 WiFi、早期的以太网）。

当多个节点同时发送帧时，信号会在接收方处“碰撞”，导致所有帧都受损。为了协调这些节点的发送行为，我们需要 多路访问协议 (Multiple Access Protocol)。

多路访问协议的分类 ​

几十年来，人们研发了数百种多路访问协议，主要分为三大类：

核心概念: 多路访问协议三大族群

  1. 信道划分协议 (Channel Partitioning Protocols)：将信道资源（时间、频率、代码）划分给各个节点。
  2. 随机访问协议 (Random Access Protocols)：不划分信道，允许碰撞，并从碰撞中恢复。
  3. 轮流协议 (Taking-turns Protocols)：节点轮流发送，通常具有更多数据要发送的节点可以获得更长的发送时间。

信道划分协议 ​

核心概念: TDMA 与 FDMA

  • TDMA (时分多路复用)：将时间划分为帧，每帧进一步划分为时隙，每个节点被分配一个固定的时隙。即使节点没有数据要发，其时隙也会闲置。
  • FDMA (频分多路复用)：将信道频谱划分为多个频段，每个节点分配一个固定的频段。

随机访问协议 ​

随机访问协议是当今局域网（如以太网、WiFi）的核心。当发生碰撞时，节点会等待一个随机时延后重发，直到帧成功通过。

核心概念: ALOHA 协议

  • 时隙 ALOHA (Slotted ALOHA)：将时间划分为时隙，节点仅在时隙开始时发送。如果碰撞，以概率 在后续时隙重传。其最大效率约为 37%。
  • 纯 ALOHA：不划分时隙，节点有数据就发。由于重传的帧更容易再次碰撞，其最大效率仅为时隙 ALOHA 的一半（约 18%）。

核心概念: CSMA：载波侦听多路访问

在发送前先“听”。如果信道忙，则等待；如果信道闲，则发送。 这类似于人类对话：在开口说话前，先听听别人是否在说话。

易错辨析: CSMA/CD 与 CSMA/CA 的核心辨析

  • CSMA/CD (碰撞检测)：用于有线以太网。发送的同时继续“听”，一旦发现碰撞，立即停止发送并发送一个拥塞信号。这极大减少了浪费在受损帧传输上的时间。
  • CSMA/CA (碰撞避免)：用于无线 WiFi。由于无线环境下节点无法可靠地进行硬件级碰撞检测（隐蔽站问题），WiFi 采用预约机制或随机退避来尽量“避免”碰撞发生，而不是去检测它。

交换局域网 ​

在现代局域网中，节点通常不直接连接到共享电缆，而是通过 交换机 (Switch) 连接。

MAC 地址和 ARP ​

虽然 IP 地址标识了主机在互联网中的逻辑位置，但链路层交换需要物理地址。

核心概念: MAC 地址

每个适配器都有一个全球唯一的 MAC 地址（又称物理地址或 LAN 地址）。它长 48 比特，通常用十六进制表示（如 \texttt{1A-23-F9-CD-06-9B}）。 类比：IP 地址像你的邮寄地址（随搬家改变），而 MAC 地址像你的身份证号（终身不变）。

核心概念: ARP：地址解析协议

由于 IP 分组需要封装在链路层帧中，发送方必须知道目的 IP 对应的 MAC 地址。 ARP (Address Resolution Protocol) 负责这种转换。

  1. 发送方在子网内广播一个 ARP 查询分组（包含目的 IP）。
  2. 子网内拥有该 IP 的主机回复一个包含其 MAC 地址的响应。
  3. 发送方将该映射存入本地 ARP 表。

以太网 ​

以太网是当今世界上最流行的有线局域网技术。它提供的服务是不可靠的且无连接的。

核心概念: 以太网帧结构

  • 前导码：用于同步收发双方的时钟。
  • 目的和源 MAC 地址。
  • 类型：指示上层协议（如 IP）。
  • 数据 (Payload)。
  • CRC：循环冗余校验。

链路层交换机 ​

交换机是一个透明的设备。

核心概念: 交换机的自学习 (Self-learning)

交换机不需要人工配置，它通过观察入站帧的源 MAC 地址来自动构建 交换表。

  • 如果目的 MAC 在表中：执行过滤或转发。
  • 如果目的 MAC 不在表中：执行泛洪 (Flood)。

易错辨析: 交换机与路由器的对比

  • 交换机：第 2 层设备，即插即用，转发速度快，但容易产生广播风暴。
  • 路由器：第 3 层设备，需要配置 IP，隔离广播域，路径选择更智能，但处理开销较大。

虚拟局域网 (VLAN) ​

现代机构的局域网经常需要将物理上分散的部门逻辑上隔离开来，或者将物理上连接在同一台交换机上的不同部门隔离开。传统的交换机无法阻止广播流量（如 ARP 请求）跨部门传播，这带来了安全隐患和性能下降。

核心概念: VLAN 的核心特性

虚拟局域网 (Virtual Local Area Network, VLAN) 允许支持 VLAN 的交换机在一个单一的物理局域网基础设施上，配置定义多个逻辑上完全独立的局域网。

  • 端口隔离：网络管理员可以将交换机的端口划分为不同的组（例如，端口 1-8 为 VLAN 1，端口 9-15 为 VLAN 2）。这两个 VLAN 在逻辑上就像是两台物理上完全断开的交换机，广播流量绝对不会跨 VLAN 传播。
  • VLAN 间路由：如果 VLAN 1 的主机需要与 VLAN 2 的主机通信，它们必须通过路由器（或具有第 3 层路由功能的多层交换机）进行。
  • VLAN 干线连接 (VLAN Trunking)：当 VLAN 跨越多台物理交换机时，我们使用一条特殊的干线端口将它们连接。为了让接收交换机知道某个帧属于哪个 VLAN，IEEE 定义了 802.1Q 标准，在原始以太网帧的首部中插入了一个 4 字节的 VLAN 标签 (VLAN tag)。

链路虚拟化：网络作为链路层 ​

核心概念: 多协议标签交换 (MPLS)

MPLS 是一种在ISP网络中广泛使用的技术。它在第 2 层（链路层）和第 3 层（网络层）之间插入了一个 MPLS 首部。 与传统 IP 路由器根据目的 IP 地址进行最长前缀匹配转发不同，支持 MPLS 的交换机根据附加的固定长度标签进行极速的硬件转发。 它最强大的能力在于流量工程 (Traffic Engineering)：网络运营商可以强制具有相同目的 IP 的两股不同流量走完全不同的物理路径，或者在链路发生故障时实现毫秒级的快速重路由。

数据中心网络 ​

诸如谷歌、微软和亚马逊这样的大型云服务提供商运营着包含数十万台服务器的庞大数据中心。数据中心网络 (Data Center Network) 将这些服务器互联，并连接到公共因特网。

核心概念: 数据中心网络架构特征

  • 机架顶 (TOR) 交换机：服务器通常堆叠在机架中，每个机架的顶部有一台 TOR 交换机，将该机架内的所有服务器互联。
  • 高度互联的拓扑：为了支持海量的“东西向流量”（即数据中心内部服务器之间的通信，如分布式计算或数据库同步），数据中心不使用传统的树状拓扑，而是采用如 Clos 网络 或 胖树 (Fat-Tree) 等拓扑，以提供极高的二分带宽和多条并行路径。
  • 软件定义网络 (SDN) 的全面应用：数据中心的网络管理和路由极度依赖集中式的 SDN 控制器，以实现灵活的负载均衡和微秒级的故障恢复。

回顾：Web 页面请求的历程 ​

为了将前面六章（应用层、运输层、网络层、链路层、物理层）的所有知识融会贯通，我们来完整回顾一个真实场景：当你在浏览器中输入网址后，究竟发生了什么？

例题: 一次完整的 Web 请求生命周期

假设一名学生 (Alice) 带着她的笔记本电脑来到学校的咖啡馆，连接校园 WiFi，并试图访问 Google 主页。

  1. 获取 IP 地址 (DHCP)： Alice 连接 WiFi 后，操作系统首先通过 UDP 广播一个 DHCP Discover 报文。该报文被封装在 802.11 WiFi 帧中，发送到 AP，再由 AP 转为以太网帧发给 DHCP 服务器。经过 DORA 过程，Alice 获得了自己的 IP 地址、默认网关 IP 和本地 DNS 服务器 IP。
  2. 解析网关 MAC 地址 (ARP)： Alice 想要向外部发送请求，必须先把数据包发给默认网关（第一跳路由器）。但她不知道网关的 MAC 地址。于是她广播一个 ARP 请求，询问“谁有网关 IP 的 MAC 地址？”。网关回复 ARP 响应，Alice 将其存入 ARP 缓存。
  3. 域名解析 (DNS)： Alice 的浏览器需要解析 \texttt{www.google.com}。它生成一个 DNS 查询报文，封装在 UDP 数据报中，目的 IP 是本地 DNS 服务器。本地 DNS 服务器通过迭代查询（访问根、TLD 和权威服务器），最终获取 Google 服务器的 IP 地址并返回给 Alice。
  4. 建立 TCP 连接 (三次握手)： 浏览器现在有了 Google 的 IP 地址。它向该 IP 发起 HTTP 请求前，必须先建立 TCP 连接。Alice 发送一个 SYN 报文段。 在途经的路由器中，这个 SYN 报文段的源 IP 被校园网边缘路由器的 NAT 转换为公共 IP 和新端口。最终到达 Google 侧，经历 BGP 路由选择和 防火墙/负载均衡器 后，Google 返回 SYNACK，Alice 回复 ACK，连接建立。
  5. HTTP 请求与响应： 浏览器生成 HTTP GET 请求，将其封装在 TCP 报文段中发送。Google 的服务器处理请求后，返回包含 HTML 数据的 HTTP 200 OK 响应。
  6. 页面渲染： Alice 的浏览器提取 HTML，解析其中的 CSS、JavaScript 和图片资源，并通过已经建立的持续连接（或开启新的连接）拉取这些资源，最终在屏幕上渲染出 Google 的搜索页面。

这个看似瞬间完成的操作，实际上瞬间调动了跨越全球的数十种协议和上百台网络设备！
